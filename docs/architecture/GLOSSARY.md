## Purpose

One word, one concept, one layer.

This glossary is the single source of truth for YARNNN terminology. Every ADR, architecture document, prompt, surface string, and marketing artifact must use the terms defined here. Drift triggers a correction PR — this is not soft guidance.

The glossary exists because YARNNN operates across four layers of cognition (the product, the role palette, the authored workers, the independent judgment seat) and three kinds of readers (users, developers, investors). Without discipline, the same word collapses layers and confuses readers. With discipline, every term lands on exactly one thing at exactly one layer.

One upstream discipline governs everything here: **semantic content lives in the filesystem** (FOUNDATIONS Axiom 1, the Substrate dimension). When a glossary term names a "ledger," a "policy," a "log," or any accumulated object, its definition must name the file path — not a DB table. Drift from file-native terminology is drift from the architecture, and both get corrected together.

## Operating System Framing (ADR-222, FOUNDATIONS Principle 16)

YARNNN is canonized as an agent-native operating system. The framing is literal — every box in OS architecture has a corresponding YARNNN artifact. The vocabulary below is canonical and replaces the would-have-been "workspace type / workspace mode" naming.

| Term | Meaning | Note |
|---|---|---|
| **Kernel** *(informal-canonical)* | The substrate layer: filesystem + primitives + axioms + privileged daemons. Domain-agnostic by construction. Equivalent to OS kernel. | Used informally alongside "substrate" (technical-canonical). "Kernel" is the macro frame; "substrate" is the technical implementation. Both valid. |
| **Syscall ABI** *(informal)* / **Primitive matrix** *(canonical)* | The set of typed verbs userspace can invoke against the kernel — the only entry point into substrate operations. Listed in `CHAT_PRIMITIVES` + `HEADLESS_PRIMITIVES` (ADR-168). | Same artifact, two framings. "Primitive matrix" is the technical name; "syscall ABI" is the OS-faithful framing. Both valid. |
| **Shell** *(informal)* | The interactive interface to the kernel — application code, not kernel. The YARNNN chat agent (`api/agents/yarnnn.py`). **ADR-407 D2 (2026-07-06): one filesystem, N shells** — each member's shell *instance* (windows, drawer, read cursors) is member-experience scope (`member_state` + per-(workspace, principal) keys); the compositor's inputs (SURFACES overlay, program declaration) remain workspace scope. | Per ADR-205, the shell is a regular `agents` table row using the same syscall surface as everything else. No privileged shortcuts. The N-shells model is macOS-not-Figma: pull freshness, CAS conflicts at save, no continuous view-sync (ADR-407 D2). |
| **Userspace** *(informal)* / **Workspace** *(canonical)* | The operator's `/workspace/` — a multi-principal attributed commons (ADR-373: one workspace = N principals — humans, agents, platforms, foreign/local LLMs — each attributed as itself), owned by its members, not by the kernel. **The workspace is the OUTERMOST unit YARNNN composes (ADR-378): one commons, served if-and-only-if an enterprise fits it; a second workspace is allowed but UNRELATED — cross-workspace federation (shared steward, org hierarchy, cross-workspace grants) is explicitly undefined, the deliberately-unbuilt layer above the ceiling.** | Both terms valid. "Workspace" is the canonical project term; "userspace" is the OS-framing word. Workspaces are not "typed" — they run programs. The ceiling is a *positive definition* ("the workspace is the unit; federation undefined"), not a headcount cap ("one enterprise max"). |
| **Program** *(canonical)* | An application that runs in YARNNN userspace. Currently: alpha-trader (built), alpha-prediction (reference SPEC), alpha-defi (reference SPEC). | Equivalent to a desktop application. Programs declare what they need; the kernel doesn't know about them. See [docs/programs/](../programs/README.md). |
| **Program bundle** *(canonical)* | The declarative package shipping a program: manifest + reference workspace + composition manifest + dependencies. Lives at `docs/programs/{program}/`. | Equivalent to `.app` (macOS), `.deb` (Debian), `.apk` (Android). Versioned in-repo. |
| **Program manifest** *(canonical)* | The top-level declaration in a program bundle (currently `README.md` at bundle root). Names what the program is, what kernel features it depends on, what surfaces it commits to, success bar. | Equivalent to `Info.plist` (macOS), `AndroidManifest.xml` (Android), `manifest.json` (Chrome extensions). |
| **Composition manifest** *(canonical, format TBD)* | The bundle-shipped declarative spec for surface composition (`SURFACES.yaml` or equivalent — exact format defined by Program Bundle Specification ADR forthcoming). | Replaces would-have-been "workspace type" / "workspace mode." A program declares its surfaces; the compositor reads. |
| **Compositor** *(canonical, infra not yet built)* | The FE/API infrastructure that resolves a program's composition manifest against the operator's workspace substrate and renders the cockpit. Reads substrate; never authors. | New architectural layer named for the first time in ADR-222. Equivalent to a window manager / desktop environment compositor. Implementation ADR forthcoming. |
| **System component library** *(canonical, partial)* | The universal FE component library (`web/components/library/`) — building blocks composition manifests reference. | Equivalent to shared system libraries (libc, libGL). Additive-only; removing a component is breaking. Convention to be formalized in implementation ADR. |
| **Workspace overlay** *(canonical, infra not yet built)* | Operator-authored deltas to a program's composition manifest, lives at `/workspace/SURFACES.yaml`. Optional; defaults to program's manifest. | Equivalent to a user's customized DE preferences. Revision-tracked under Authored Substrate. |
| **Distribution** *(informal)* | A program understood as a curated bundle of (composition manifest + reference substrate + dependencies + commitments) on top of the YARNNN kernel. Same kernel; different program → different distribution. | Informal aid for positioning. Not a canonical technical term — programs and bundles are. |
| **Activation ladder / Rung 0·1·2** *(canonical, ADR-level — NOT an axiom)* | The three activations of a workspace, the granularity ADR-380 refines ADR-375's binary `AGENT_ENABLED` into. **Rung 0** = the substrate wedge (`remember`/`recall`/`trace` + connectors + files; steward absent — `AGENT_ENABLED=false`; reads + attributed *dumps*, never placed). **Rung 1** = Freddie, the 1st-order substrate steward (derive-and-cite, placement, multi-principal arbitration, persona-agent governance; reasons as a base-LLM; substrate-internal mutations that are **reversible**; ships on engineering time). **Rung 2** = 2nd-order persona agents taking **consequential external action** (capital, irreversible sends) under an autonomy grant — gated by an **exogenous track-record clock**, deferred from the launch build. | The launch **deferral line** is the Rung-1→Rung-2 boundary = *autonomy over consequential action* = the ADR-307 consequential gate — **not** "judgment" and **not** "steward on/off." **Harness honesty** (ADR-380 D3): budget/pace are live on Freddie (Rung 1); mandate/autonomy go *degenerate* on a stakeless steward — autonomy-harness validation accrues only at Rung 2. **NOT an axiom** (ADR-380 §6 — an ADR-level granularity; promoting it to an axiom is a separate deliberate decision). See [ADR-380](../adr/ADR-380-the-activation-ladder-and-the-judgment-deferral-line.md). |
| **Management plane** *(canonical, ADR-338)* | The operator-facing projection of the four flows: the standing set of surfaces through which the operator *owns* the operation — source/watch management (context-in), deliverable specs + preferences (work-out), attestation/import/verdicts (outcomes-in), delegation/budget/calibration dials (the loop). First-class product surface, never "setup overhead to automate away." | The OS framing's **experience half** (Derived Principle 28): the operation is the running system; the management plane is System Settings + Finder + the installer + the permission dialog. Channel-dimension projection rendered by ADR-245's L1/L2/L3 model — no new architectural layer. Governed by the **consent line** (below). |
| **Consent line** *(canonical, ADR-338)* | The discipline separating surface-worthy acts from invisible automation: acts that change *what the operation is allowed to do or perceive* (activate a program, bind a transport, declare watches, grant delegation, set budget, approve a diff, attest an outcome) get first-class surface; acts that change *what the substrate currently says* within granted allowances (fork, fetch, distill, mirror, materialize) stay invisible under the revision chain. | Prevents the management-plane reframe from rotting into ceremony: legible-and-owned, not laborious. A driver install got simpler over an OS's life but never invisible-without-consent. Per Derived Principle 12, hiding consent moments caps the autonomy an operator will ever grant. |
| **App Store / installer / drivers** *(informal-canonical, ADR-338)* | The experience-register names for management-plane surfaces: **App Store** = program catalog + activation (Settings→Workspace per ADR-244, `/setup` flow-walk per ADR-331); **installer** = the activation sequence surfacing a bundle's four-flow declaration before fork; **drivers** = transport bindings the operator grants (platform connectors, watch sources via `_sources.yaml`, future MCP bindings per ADR-335 Crawl-B). | Operator-ratified surviving vocabulary (2026-06-11). Pairs with the system-side rows above: kernel/programs (system half) ↔ App Store/drivers/Settings (experience half). |
| **Mirror surface** *(canonical, ADR-340)* | A surface fronting exactly one substrate concern — complete, neutral, faithful (sources, autonomy, budget, files, activity, recurrence…). The experience-level escape hatch; the `/proc` of the OS. Every substrate concern earns exactly one mirror (ADR-297's discipline, named). | Mirrors are never deleted and never reworked into opinions — and they are **not the operator interface**: a launcher that foregrounds mirrors is L1-weighted at the experience level (ADR-245's L1/L3 lifted one level, Derived Principle 29). |
| **Composition surface** *(canonical, ADR-340)* | A surface fronting one operator-**act** — selective, opinionated, synthesized over many substrates, program-weighted under the ADR-312 contract (kernel owns the slots; programs weight/label/shape; kernel never invents a program noun). Home is the first instance; the act-set defines the full set. | Compositions foreground; mirrors are reachable from compositions, never the default. A mirror may *coincide* with an act (Queue↔Decide, Files↔artifacts) — coincidence is what earns a mirror primary placement. |
| **Standing loop (operator acts)** *(canonical, ADR-340)* | The operator's recurring act-set: **Decide** (consent moments) · **Read** (what happened) · **Dwell** (where it stands) · **Tune** (granted allowances) · **Amend** (constitution) · **Setup** (pass-through). Derived from the four flows × the consent line — never from a persona. | The macOS resolution to unbounded user-shapes: the OS commits to few universal acts; programs carry the use-cases. The act-shape is kernel; the act-content is program. |
| **Attention routing** *(canonical, ADR-340)* | The OS responsibility of bringing demand to the operator — **derived from substrate, never stored** (pending `action_proposals` + narrative weight taxonomy + budget runway; a `notifications` table is the ADR-153 shape of mistake). Three channels: badges/menu-bar vitals (state-at-a-glance), the **attention center** (top-bar aggregation, pure deep-links, no content of its own), in-the-moment dialogs (operator present). **ADR-410 (2026-07-06) makes the derivation singular + per-viewer**: two questions, two sources — "what wants me" = the witness queue (`action_proposals`), "what happened" = the workspace timeline (the three attributed ledgers) filtered per viewer (self-acts excluded at the bell per ADR-405 D4). Bell = glance, Notifications = workbench, Home slot = ambient — three depths, one source. | Consequence: mirrors stop being attention destinations — the operator arrives at the Queue by routing, not by remembering. Distinct chrome roles: the status cluster is Control Center (standing state); the attention center is Notification Center (events). The chat-narrative-derived activity feed is retired (ADR-410 §1 — post-ADR-407 the chat thread is the viewer's private session). |
| **Invocation contract** *(canonical, ADR-413)* | The kernel-invariant layer every external-engine invocation satisfies, regardless of protocol or engine: acts under a **principal + grant + gate** (DP23), consumes inputs only as **DP34 projections**, produces outputs only as **attributed revisions**, and meters on the **one ledger** (a rate-row exists before an engine is invokable). DP34's Mechanism-dimension sibling: DP34 fixes what crosses *into* a model; this fixes what any invocation *is* to the kernel. | Layer ① of ADR-413's four-layer stack (contract → drivers → catalog → intent vocabulary). Never code-forked per engine — the contract is what makes engines swappable data. |
| **Protocol driver** *(canonical, ADR-413)* | The ONLY code seam in the external-engine stack: one driver per **interaction protocol** — the physics/economics-derived classes (~4): **tool-loop engine** (LiteLLM is this class's driver), **async job** (the render service is the in-house precedent), **stateful session**, **sync query**. The Unix driver-class axiom applied outward: I/O discipline, never content type — modality is a *payload property* that dissolves at the engine level (ADR-413 v2's re-derivation; "language vs image capability" rejected as the seam). | ADR-335 made the same cut inbound (transports as peripherals, known by contract). Anti-goal: never force an async-job or session engine through the tool-loop protocol. New drivers are demand-gated, each its own ADR (ADR-413 D5). |
| **Engine catalog** *(canonical, ADR-413)* | Engines are **data, never code**: catalog rows carry the engine id, its protocol class (which driver serves it), payload tags (modalities — plural per engine), and its rate row (`_BILLING_RATES` — a rate-row-before-entry is the metering precondition). Generalizes ADR-402 (kernel model routing as data) + ADR-379 (host profiles) + ADR-411's `LANE_MODELS` whitelist. | Catalog unification is gated to the first non-tool-loop driver's ADR — until then the three registries stay separate (honest deferral, ADR-413). |
| **Intent vocabulary** *(canonical, ADR-413)* | Where the word "capability" survives: the UX-layer naming of what an operator can *ask for* ("draft an image", "transcribe this") — resolved to an engine + driver at invocation time. Never a code boundary, never a driver seam. | Layer ④. Engine choice is product law at Altitude 2 (recommend-never-conceal, the member picks); INVERSE at Altitude 1 (the steward's engine is kernel-chosen, ADR-402). |

**Retired vocabulary candidates** (rejected as housing for the specialization layer):

- **Workspace type / workspace kind** — rejected by ADR-222. Workspaces don't have types; they run programs. The kernel must not know about specific programs (kernels don't know applications). Type tags would have required kernel branches; the OS framing makes them unnecessary.
- **Workspace mode** — rejected by ADR-222. Implies switchability that doesn't exist (a trader workspace doesn't switch into a commerce workspace at runtime), and conflates Identity with Channel.

**Usage discipline:** when introducing a new mechanic that touches the architectural layering, name which layer it occupies (kernel / compositor / program-bundle / userspace / shell). A mechanic that crosses layers without justification is a design error per Derived Principle 16.

## Operation Rate, Output Contract & Witnessing (the operator's standing declarations)

> Every workspace is specified by four orthogonal operator declarations. Two are about *how the operation runs* (Rhythm, Witness dial), one is about *what it owes* (Expected Output), one is about *how it reasons* (Persona). The load-bearing discipline: **Rhythm and Expected Output are fundamentally different and must be captured separately** — a rate is not a deliverable, and neither derives from the other.

| Term | Meaning | Note |
|---|---|---|
| **Rhythm** *(canonical, ADR-327)* / **Heartbeat** *(informal)* | The operation's **rate of attention** — how often it wakes, reasons, and spends a judgment cycle. Declared as a spend envelope (`_budget.yaml`: `amount_usd × window`); tempo IS the spend (ADR-327: *"pace was always a budget wearing a frequency costume"*). Governance root — the operator declares it; the Reviewer allocates wakes within it, never authors it. | Answers *"how often do I show up to work?"* The Trigger-dimension dial of the operator trifecta (Rhythm · Witness · Persona; ADR-298 D11). `_pace.yaml` retired (ADR-327) — there is no separate pace dial; spend-over-time is tempo. |
| **Expected Output** *(canonical, ADR-345)* | The operation's **output contract** — what it is on the hook to *produce*: the *kind* of artifact, the *delivery-cadence*, and the *bar*. The measurable half of the MANDATE (*why we exist*, made concrete). Declared as prose in `constitution/MANDATE.md` → `## Expected Output` (the operator's promise, survives occupant rotation) with a machine sidecar `_expected_output.yaml` the standing-obligation check (Derived Principle 30) reads. | Answers *"what am I on the hook to have produced?"* **Orthogonal to Rhythm** — a trader can wake every minute and correctly produce zero trades; an author can wake weekly and owe two essays a month; neither derives from the other. A **delivery-cadence + bar, floor-gated — never a hard volume quota** (a quota pressures the floor toward marginal output; a cadence the floor regulates). |
| **Witness dial** *(canonical reframe, ADR-345; mechanism ADR-307)* / **Autonomy** | The dial that selects **which consequential beats the operator witnesses before they bind** — NOT whether the agent works. The agent always works the full job (it is a judgment seat acting in absence); the dial routes attention. `autonomous` = the whole rhythm runs subconsciously (operator reads the trail at leisure); `bounded`/`manual` = chosen beats surface first (route to `action_proposals`). The Mechanism-dimension dial of the trifecta. | The mechanism is already this (`permission.py`: the gate runs *after* the Reviewer decides; QUEUE = *acted-and-waits-for-witness*, never *blocked*). ADR-345 flips the **prose** from the prior "trust ceiling / approval degree" framing (which described the operator's permission-anxiety, not what the agent does). Resolves "does full autonomy mean zero operator approval?" → **yes: the agent works the whole job; the dial only decides what reaches the operator's attention.** |

**Usage discipline:** when a design surfaces a "should the operator decide X?" question, first ask whether X is a *witness* question (which the Witness dial answers — does this beat surface?) or a *contract* question (which Expected Output answers — what does the operation owe?). A Reviewer that Clarifies "what should I produce / on what cadence?" when Expected Output is undeclared is surfacing a **missing-contract symptom**, not correct consent-seeking (ADR-345; the 2026-06-18 author Path-A/B Clarify).

## The Three Scopes & The Viewer (ADR-407 + ADR-412 D6, FOUNDATIONS Derived Principle 35)

Every persistent store declares exactly one scope; ADR-407 §3 is the living scope registry. The viewer is a **parameter of rendering, never a fork of composition**.

| Term | Meaning | Note |
|---|---|---|
| **Workspace content** *(scope, canonical ADR-407)* | The commons — keyed `workspace_id`: substrate + revisions, the three act ledgers (`workspace_file_versions` / `execution_events` / `action_proposals`), grants, balance. Every member reads the same content. | The re-key ADR-373 started; ADR-407 Phases 0–5 completed the read/enforcement half. |
| **Member experience** *(scope, canonical ADR-407)* | One member's private operating state — keyed `(workspace_id, principal_id)`: chat sessions + lanes, shell/window state, read cursors. **Never consulted for authorization; never holds authored content** — the two invariants that keep it from becoming a shadow substrate. | Home: the `member_state` table (`GET/PUT /api/member-state/{key}`) + per-(workspace, user) localStorage keys as its cache (ADR-407 Phase 3). |
| **Account** *(scope, canonical ADR-407)* | The human across workspaces — keyed `user_id`: credentials, billing identity, theme. | The UserMenu "Account" window (ADR-347) is its surface. |
| **Member state** *(canonical, ADR-407 Phase 3)* | The server-backed member-experience store: `member_state(workspace_id, principal_id, key) → jsonb`, write-through from the shell (debounced, local-wins), newer-of reconciliation on mount. The attention read-cursor lives here. | Presentation state, not workspace state — DP29's derived-never-stored applies to the *content* of attention; the cursor is the one timestamp per (workspace, principal) it permits. |
| **Viewer parameter** *(canonical, ADR-412 D6)* | The rule that one commons renders **first-person per viewer** via derivation, never per-viewer copies or member-shaped surfaces: attribution resolves "You" vs the peer's name (`resolveActorForViewer` over the roster + ledger identity); attention derives per viewer (ADR-410: one derivation, N mounts — self-acts excluded at the bell per ADR-405 D4); **affordances render per the viewer's grant coverage (`write_regions`), never a role enum** — the UI twin of ADR-405's no-species-law. Reads stay universal; FE gating is legibility, the server gate is enforcement. | FE home: `web/lib/workspace/viewer.ts` (roster + memberships module-cached; `useViewerGrant`) + `GrantGate`. A fresh member lands on the same `/home`; there is no member onboarding fork, ever. |
| **Ambient commons context** *(canonical, ADR-412 D6)* | The always-glanceable which-workspace indicator (top bar, rendered for **every** caller once the membership read resolves — N=1 shows "My workspace"; operator ruling 2026-07-06: consistent chrome over conditional chrome) + the compact who's-here roster read (**membership, never presence** — the ADR-373 no-realtime rejection stands). Depth lives at Workspace Settings → Workspace Members. | FE: `WorkspaceIndicator` (chip + popover); switching stays in the UserMenu (one switcher, ADR-407 Phase 5 — the SWITCHER stays >1-gated: switching needs alternatives, naming doesn't). |

## Three Altitudes, Three Chromes (ADR-408 D2 + ADR-412)

The three kinds of AI in a workspace, each with exactly **one chrome home** — *placement is the pedagogy* (same chrome must never imply same kind; no labeling fixes what placement asserts).

| Altitude | Kind | Chrome home | Never appears as |
|---|---|---|---|
| 1 | **Freddie** — the system agent (ADR-381) | **The rail only** (the chat drawer, ADR-316): part of the shell, summonable everywhere. Inspection panes live at Workspace Settings → **System Agent** (ADR-412 D5, reversing ADR-387 §6.4). | a window, a launcher tile, a roster card, a chat among chats |
| 2 | **Lanes** — the member's model-pinned helpers (ADR-411) | **The Chat surface** (`/chat`, ADR-412 D3): a windowed workbench of the member's conversations, work-first recents (model = chip + filter facet, never folders — ADR-412 D4 on the ADR-385 group-by-relationship precedent). | chips beside the steward; a principal |
| 3 | **Agents** — persona agents + user-authored domain agents | **`/agents`** — the judgment roster (Altitude 3 only; the governor frame line names Freddie without seating him, ADR-381 D5). | conflated with either of the above |

Cross-references are links, not co-presence. The Chat surface is a *workbench over the shared workspace*, never the product's center — Home stays the front page; the ADR-411 contract holds (lanes are isolated conversations; the workspace is the shared memory).

## The Four Flows (ADR-332, FOUNDATIONS Principle 26)

An operating workspace runs four flows; a program is a flow-declaration set. This vocabulary is the diagnosis-and-authoring language for operation completeness.

| Term | Meaning | Note |
|---|---|---|
| **Four flows** *(canonical)* | The directional loop an operating workspace runs: **context in** (perception) → **work out** (the acts) → **outcomes in** (reality's verdict on the operation's own acts) → **the loop** (calibration). | The *loop view* of one operation cycle. Composes with Axiom 2's perception layers (the *source taxonomy*) without conflict. Outcomes-in is the self×world coupling term — why Axiom 8's substrate is the moat's spine. |
| **Flow-completeness** *(canonical)* | All four flows declared and runnable (or a flow explicitly N/A with rationale). Bundle-conformance requirement for active programs (first instance ADR-330 D4; general gate at arc-3 per ADR-287). | The remedy for incompleteness is a declaration (bundle-side) or an intake/transport (kernel slot) — never a parallel subsystem. |
| **Flow-incompleteness** *(canonical)* | The structural name for generic-workspace "offness" — one or more flows absent/undeclared. Supersedes looser explanations (empty substrate, cold start) — those are symptoms. | Diagnostic test: enumerate the four flows; the missing one is the diagnosis. |
| **Perception field** *(canonical)* | Flow 1's world-present cell: what the operation *watches* — a declared universe, read on cadence, **distilled** into attributed signal substrate, pruned by calibration over tenure. Three layers with distinct durability classes: **Declaration** (kernel — judgment, cannot commoditize) · **Observation contract** (kernel — the form reality takes) · **Transport** (commodity — swappable drivers). | Discipline fixed by ADR-332 D5; built by [ADR-335](../adr/ADR-335-perception-field.md) (Crawl-A 2026-06-11: `substrate_abi.watches` slot + observation contract + D9 four-flow gate; MCP client at Crawl-B, demand-pulled). Reading the world is commodity; the declared/distilled/tenured/calibrated field is the asset (FOUNDATIONS Derived Principle 27 + Axiom 1 eighth sub-clause). |
| **Watch** *(canonical, ADR-335)* | A Layer-1 perception declaration: WHAT the operation watches (`shape` — a perception shape, never a vendor), WHERE the operator edits it (`declaration` — watch substrate, e.g. alpha-trader's `_universe.yaml`), WHICH recurrence enacts its cadence (`recurrence` — the Trigger pointer; cadence lives ON the recurrence, singular), WHERE observations distill to (`distills_to` — signal substrate). Declared in `substrate_abi.watches`; read via `bundle_reader.get_watches_for_workspace` (the perception twin of `get_ground_truth_for_workspace`). | The cadence dial spans once-to-forever: a harvest (ADR-331 Mode-A) is a watch with a `once` Trigger; graduation between them is the perception instance of the inline-action → task gradient (Axiom 9 Clause C). Watch-first, transport-second (ADR-335 D5): a transport enters only because a declared watch needs it. The watch portfolio is calibrated by flow 4 over tenure. |
| **Observation** *(canonical, ADR-335)* | The only form reality enters substrate: `{watch_id, source_ref, attestation (platform\|operator\|agent), observed_at, distilled_content}` — **distilled**, never raw-mirrored (ADR-153). Written to attributed signal substrate via ADR-209. | The perception twin of `OutcomeCandidate` (ADR-330); reuses its attestation enum verbatim. ### Two compose modes (post-OS-framing discourse, 2026-04-27)

The compose substrate has two modes. Both are kernel-level; both share the universal component library; both are program-aware via composition manifest. They differ in lifetime and binding mode.

| Term | Meaning | When it applies |
|---|---|---|
| **Document compose** *(canonical)* | Produces frozen HTML artifacts (`output.html`, PDF exports, PPTX). Snapshot at compose time, immutable thereafter. | Task pipeline output composition, emailable deliverables, archived task runs. Existing infrastructure (ADR-148, ADR-170, ADR-177, ADR-213). |
| **Surface compose** *(canonical)* | Produces live Home slots — re-rendered every load, bound to current substrate. | The Home's program sections displaying live workspace state, dashboards, metric bands (ADR-312; renamed from "cockpit panes"). |

A `MetricCardRow` component works in both modes — call site decides whether the binding is live (surface compose) or snapshot-at-compose-time (document compose). Mode is a property of the binding in the SURFACES manifest, not the component.

Surface compose can *embed* document compose artifacts (e.g., Overview's daily-discipline band uses a `TaskOutputViewer` component that reads the stored `output.html` from the latest daily-update run). This is normal — a surface arranges components, some of which display frozen artifacts.

## Workspace Permission Topology (ADR-320, FOUNDATIONS Derived Principle 25)

For the **multi-principal commons** (ADR-373), permission is now **defaulted-by-meaning, owned-by-grant**: the meaning-folder supplies a *create-time default grant* stamped onto the new file; the **file's own grant** is the runtime authority (the gate reads the file's grant for edit/delete, not the prefix). ## The Six Dimensions (FOUNDATIONS Axiom 0)

Every mechanic in YARNNN occupies a cell in six orthogonal dimensions. These are the axiomatic vocabulary that governs every other term in this glossary. Each dimension answers one interrogative:

| Term | Interrogative | Decides | Canonical reference |
|---|---|---|---|
| **Substrate** | What | What persists between invocations | FOUNDATIONS Axiom 1 (filesystem) |
| **Identity** | Who | Which cognitive layer acts or authors | FOUNDATIONS Axiom 2 (four layers) |
| **Purpose** | Why | What intent drives the work | FOUNDATIONS Axiom 3 |
| **Trigger** | When | What invokes execution (periodic / reactive / addressed) | FOUNDATIONS Axiom 4 |
| **Mechanism** | How | By what means — spectrum from deterministic code to LLM judgment | FOUNDATIONS Axiom 5 |
| **Channel** | Where | To what location or surface output is addressed | FOUNDATIONS Axiom 6 |

**Usage rule:** when writing an ADR, a design doc, or a code comment that introduces a new mechanic, state explicitly which dimension(s) it occupies. A mechanic that spans dimensions without explicit justification is a cross-cut — cross-cuts are legitimate only when argued for (e.g., compose substrate deliberately couples Mechanism + Channel per ADR-148). See FOUNDATIONS v6.0 Derived Principle 1 (Dimensional purity).

|---|---|---|
| **Wake** | The irreducible unit of agent activity — *something changed, judgment is warranted, now*. ## Vocabulary note: Agents, Orchestration, and Persona (ADR-216 2026-04-24)

**Agent** in YARNNN means a **persona-bearing judgment entity**: holds standing intent on behalf of a principal (the operator), reasons from an operator-authored **persona** (IDENTITY.md), applies an operator-declared **framework** (principles), renders judgments or accumulated domain expertise that carry fiduciary weight. Persona is read at reasoning time and shapes decisions. **Sharp, principal-agent sense.** Members today: **Reviewer** (sole systemic persona-bearing Agent) + **user-authored domain Agents** (instance, zero-to-many). Future systemic: Auditor, Advocate, Custodian, other judgment archetypes.

**Feed surface** (canonized by ADR-259, 2026-05-08): the operator's primary view — a chronological, multi-actor, continuously-updating timeline of everything that happens in the workspace. Operator messages, Reviewer decisions, System Agent execution narrations, recurrence completions, autonomy events all land on the feed. The operator opens the feed, scans recent activity, types when they want to direct or check in. The feed updates whether the operator is watching or not. Replaces "chat surface" as the concept name. Backend `routes/feed.py`. Frontend `FeedTimeline`, `FeedSurface`. Composer (the typed-input box) is one entry mode on the feed; the surface itself is multi-source, not request-response. **ADR-370 (2026-06-25) folded the Feed into the boundary surface as its Flow lens; ADR-385 (2026-06-29) renamed that surface `context` → `channels`** — the renderer, the typed-row grammar (ADR-289), and the `session_messages` substrate are unchanged; only the surface's home moved (the standalone `/feed` route is a redirect stub → `/channels?channels.pane=flow`). The narrative is now reached as **Channels → Flow** and, redundantly, **Notifications → Activity**.

**Channels (surface)** (canonized by ADR-370 as "Context", 2026-06-25; **renamed `context` → `channels` by ADR-385, 2026-06-29** — "context" was ambiguous with the filesystem [Files surface] and the `operation/context/` substrate namespace): the operation's **perception + principal** surface — the edge where the workspace meets the outside world. A `SettingsPaneShell` split-nav, two groups: **CHANNELS** — what crosses the edge: **Connections** (platform data-feeds — status·coverage·freshness), **Sources** (standing web/RSS watches, ADR-335/336), **External Agents** (MCP / external-LLM principals — a *filtered view* of the `principal_grants` member roster, `role ∈ {foreign-llm, a2a, platform}`; ADR-373 + ADR-385 D3, one substrate two views per DP29). **ACTIVITY** — the running record of crossings: **Flow** (the complete narrative — FeedSurface; the *default* landing pane), **In** (inbound crossings — FeedSurface filtered by `isInbound`), **Out** (the operator-addressing dispatch ledger via `GET /api/emissions`, read-only — a *distinct* data source, not a narrative filter, which is why In/Out/Flow stay distinct nav items not filter chips). A *composition* (one surface ↔ one act, ADR-340 DP29), not a mirror. URL `/channels` (`/context` + `/feed` are ADR-308 redirect stubs). Two orthogonal planes co-locate in CHANNELS without overlap: Connections = *what feeds* the operation (`platform_connections`); External Agents = *who can write* the commons (`principal_grants`). Narrative redundancy with Notifications → Activity is deliberate tiered redundancy (ADR-367 D3 macOS principle). **Disambiguation**: the route slug is unrelated to the retired `context/` substrate root (ADR-320 → `operation/`) or "context domains" (ADR-151 fiction).

**YARNNN** is the product brand and the voice of the orchestration entry mode on the feed surface. In the cockpit, the conversational surface entity is called the **System Agent** (see below). On the feed, the system speaks as "YARNNN" — the brand and the surface entry mode share a name. YARNNN (reclassified by ADR-216): the **orchestration feed surface entry mode**, *not* a persona-bearing Agent. Platform-fixed voice (no workspace-authored IDENTITY file), no standing intent, no fiduciary weight. Retains an `agents` table row with `role='thinking_partner'` as implementation substrate for session state; the row is implementation, the classification is architectural.

**System Agent** (named by ADR-251 as cockpit entity, **dissolved at the cockpit-entity layer by ADR-272, 2026-05-14**): the brand label for orchestration **ambient activity** — regex execution router output, Reviewer-directed action narration, mechanical recurrence completions. The audit producing this reversal showed System Agent had no LLM identity backing — it was a label across three plumbing places, none of which is an LLM. Post-ADR-272, System Agent does NOT have a roster card on `/agents`, does NOT have a detail surface at `?agent=system`, does NOT speak as a chat participant. The `system_agent` chat role label collapses into an ambient `system-activity` rendering shape (Phase 2 FE). The `meta-cognitive` DB row persists as chat-mode LLM substrate referenced by `routes/feed.py` for the orchestration profile, but is no longer cockpit-visible as a peer entity. YARNNN remains the brand + chat voice. Internal DB slug `thinking_partner` and class enum `meta-cognitive` persist (glossary exceptions — data-compat, not surfaced).

**Orchestration** in YARNNN means production machinery: task pipeline, dispatch routing, capability bundles, back-office scheduling, **and the YARNNN feed surface**. Orchestration has no standing intent, no fiduciary relationship, no accumulated identity of its own. It is stateless computation (or platform-fixed-voice conversation) that runs *under* Agents. **Orchestration is never persona-bearing.** Performance-fungible (swap one orchestrator for another and work executes the same way; swap one persona-bearing Agent for another and outcomes diverge — that divergence is what persona is).

**Persona** in YARNNN means the operator-authored judgment character embodied by a persona-bearing Agent in its `IDENTITY.md`. The *character* that applies the framework. Swappable per workspace — one workspace's Reviewer can embody Simons; another's can embody Buffett; another's can embody a founder-authored original; all share identical seat structure and produce distinct output distributions. **Persona is the axis on which Agents self-improve** — calibration data in `decisions.md` and `_money_truth.md` accumulates *against the persona that produced it*, not against a generic Agent identity.

The three classes in one sentence: *Agents hold intent and render judgment through their authored persona. Orchestration runs under them, bundles capabilities, dispatches production, and surfaces the operator-facing chat. Persona is the operator-authored character that distinguishes one Agent instance from another.*

Industry's "agent" vocabulary (LLM + tools + loop + memory = "agent") maps most closely to YARNNN's **production roles** (Researcher, Analyst, Writer, etc.) — packaged production capabilities, not Agents. Industry has consolidated the word around the wrong referent. **YARNNN does not follow.** The word belongs to judgment-bearing entities and is used accordingly, in canon, code, and ADRs going forward.

External product vocabulary happens to align naturally: the "Agents" the operator sees on `/agents` ARE Agents in the sharp sense (user-authored domain workers hold operator-authored intent, represent the operator, accumulate). No external vocabulary shift required.

Retired terms (see Retired-terms table):
- "Specialist" as an entity — retired for the orchestration-role concept. Use "production role" instead.
- "Platform Bot" as an entity — retired. Use "platform integration" instead. (ADR-207 P4a already dissolved them as agent class.)

Full authoritative taxonomy: [LAYER-MAPPING.md](LAYER-MAPPING.md). Full treatment: [THESIS.md](THESIS.md) §"Vocabulary: Agents and Orchestration".

## Enforcement rule

1. **New ADRs must use glossary terms.** If a concept requires a word the glossary doesn't have, add it here first, then use it in the ADR.
2. **Retired terms cannot appear in active docs or code comments.** They may persist as internal DB slugs (e.g., the `thinking_partner` role value) where migration cost exceeds reader benefit — those exceptions are listed in the Exceptions table.
3. **Renames propagate in a single commit.** When a term is added, changed, or retired, every active doc and every prompt must move in the same PR. No staged rollouts.
4. **User-facing strings must pass the instinct test.** A non-technical user reads the term and immediately knows what it means. If a term needs a definition before it's useful, it's the wrong term.

| **Authored Substrate** | The architecture that completes FOUNDATIONS Axiom 1 by making every substrate mutation attributed, purposeful, and retained. Every write to `workspace_files` lands a new **revision** with an `authored_by` identity and a message; prior revisions are never silently lost. Covers *all* workspace files uniformly — no Postgres-vs-git bifurcation, no per-path exceptions. Three of git's five core capabilities adopted (content-addressed retention, parent-pointer history, authored-by attribution); branching and distributed replication deliberately deferred. | Not "git-backed storage" and not "file versioning." It is the substrate-level enforcement of Axiom 2 (every file has an author) and the fine-grained enabler of Axiom 7 (revision-by-revision recursion). See ADR-209 and [authored-substrate.md](authored-substrate.md). |
| **Revision** | A single attributed write to a substrate path. Carries `authored_by`, `message`, `parent_version_id`, and a content hash. The unit of Authored Substrate. | Chosen deliberately over "commit" (avoids git-implementation connotation) and over "version" (avoids collision with the legacy `workspace_files.version` integer column being retired). When writing prose, prefer *"the most recent revision of MANDATE.md"* over *"the current version"*. |
| **Revision chain** | The parent-pointered sequence of revisions for a single path. Walked backward for history traversal, diffed between any two points, reverted by repointing `head`. | A revision chain is per-path, not per-workspace. Different files have independent chains. |
| **Head** | The current revision a path resolves to. `workspace_files.head_version_id` points at it. Reads default to head unless a specific revision is named. | Not a git branch head — there is only one head per path under Authored Substrate. Multiple divergent heads would require branching, which is deferred. |
| **Authorship trailer** | The `(authored_by, message)` metadata on every revision. Required at the write-path boundary — writes without an `authored_by` value are rejected. | Replaces the ad-hoc attribution conventions previously scattered across `reviewer_audit.py` headers, the `<!-- inference-meta -->` HTML comment's author field (ADR-162), and git-commit-trailer framing that the withdrawn ADR-208 v1 had proposed. |
| **Recurrence mode** | The schema field on every recurrence record (per ADR-263) that declares at authoring time whether the recurrence wakes the Reviewer. Two values: `judgment` (recurrence's `prompt` is invoked as a Reviewer message envelope, today's behavior) and `mechanical` (recurrence's `prompt` names a primitive invocation `@primitive: ...`; dispatcher parses and executes deterministically; no Reviewer wake). Default at parse time is `judgment` (preserves backward compatibility for legacy entries). Authored at create-time by operator-via-YARNNN, Reviewer-mid-loop, or system-bundle-fork; editable thereafter via `Schedule(action="update")`. | Authoring intent is the wake signal — the recurrence's author knows whether its work warrants judgment, so the dispatcher honors the declared mode rather than deriving wake-worthiness from substrate writes after the fact. The field is the singular mechanism by which scheduled work declares its character; replaces the three-trigger taxonomy's `scheduled` sub-shape (which conflated mechanical and judgment work). See [ADR-263](../adr/ADR-263-recurrence-mode-mechanical-vs-judgment.md). |
| **Substrate-canonical world** | The axiomatic property (FOUNDATIONS Axiom 1, third clause per ADR-264) that the workspace's substrate is the workspace's complete model of every piece of state the Reviewer reasons about — internal AND external. External-system state (broker positions, fills, account, commerce revenue, news, etc.) is mediated into substrate by deterministic Python primitives invoked through `mechanical`-mode recurrences. Judgment reads substrate; it does not call external APIs as primary perception. | A design principle, not a metric. The reason `SyncPlatformState` exists. Closes the same Axiom 1 violation pattern that ADR-153 (`platform_content` sunset) and ADR-195 (money-truth substrate) corrected for other parallel substrates — external data was *de facto* allowed to live in LLM context windows because no mediation mechanism existed. ADR-264 supplies the mechanism. |
| **`SyncPlatformState` (primitive)** | The canonical mechanical-recurrence-friendly primitive (per ADR-264). Wraps `(platform-tool call → substrate write → diff-awareness)` as one atomic deterministic operation. Inputs: `tool` (platform tool name), `tool_args`, `write_to` (substrate path template), optional `iterate_field` + `item_key` for per-item iteration, `diff_aware` (skip write if content unchanged). Writes substrate via `write_revision()` with `authored_by="system:sync-platform-state"`. Registered in `HEADLESS_PRIMITIVES` + `REVIEWER_PRIMITIVES` + `HANDLERS`. **NOT in `CHAT_PRIMITIVES`** — operators don't invoke directly; they author recurrences (via Schedule) that invoke `SyncPlatformState` mechanically. | The first concrete primitive designed for use in mechanical-mode recurrences (ADR-263). Same API client as the LLM-callable platform tool surface (D4 dual surface) — different consumption pattern. The primary external-state-mediation layer per substrate-canonical-world principle. |
| **Substrate sensor** | The dispatch shape: a `mechanical`-mode recurrence whose `prompt` invokes a substrate-mirroring primitive (today: `SyncPlatformState`). Cron-fired, deterministic Python execution, writes substrate, no LLM. Operator authors per workspace; bundle ships defaults per program. | Not a separate primitive type — "substrate sensor" names the dispatch composition (mechanical recurrence + mirroring primitive), not a new code entity. Useful operator-facing vocabulary for "the things that keep my substrate fresh." |

## Verbs

What actors do. Different actors get different verbs when the cardinality or permanence of the act differs.

| Verb | Actor | Object | Cardinality |
|------|-------|--------|-------------|
| **Create** | User (through conversation with YARNNN) | an Agent | One-shot. Produces persistent identity. |
| **Draft** | YARNNN | a Team | Per-task. Iterative. Re-drafted every task cycle from the Specialist palette. |
| **Evolve** | Feedback loop | an Agent or a Specialist | Continuous. Agents evolve domain identity; Specialists evolve role-scoped stylistic memory. |
| **Scaffold** | System | a workspace | Once, at signup. Reserved for workspace-level setup. Never used for agent creation. |

**Asymmetry is deliberate.** The user *creates* an Agent (generic, neutral, universally understood). YARNNN *drafts* a Team (precise about the iterative per-task selection nature of the act). Collapsing these to a single verb loses information.

## Identity layers

What develops, and where.

| Layer | Scope | Substrate | Developer |
|-------|-------|-----------|-----------|
| **Workspace identity** | YARNNN-scoped (the user's work context) | `/workspace/IDENTITY.md`, `/workspace/BRAND.md` | User (via YARNNN, usually through inference on uploaded documents) |
| **Specialist memory** | Role-scoped stylistic preference | ADR-117 distillation artifacts (`style.md`), role-keyed | Feedback loop across all tasks that used the specialist |
| **Agent identity** | Domain-scoped, user-created | `/agents/{slug}/AGENT.md` + accumulated Domain context the Agent is responsible for | User (initial creation via conversation) + feedback loop (evolution) |
| **Reviewer identity** | Workspace-scoped, one per workspace | `/workspace/review/IDENTITY.md` + `principles.md` (declared framework, user-editable) + `decisions.md` (rolling append-only log) | System (scaffolded at signup) + User (edits principles.md) + Reviewer itself (appends to decisions.md on each review) |

The split matters because **the four layers develop along different axes**. YARNNN develops upward (better orchestration judgment). A Specialist gets better at style and preference. An Agent gets better at domain knowledge. The Reviewer's development axis is judgment calibration — approve/reject accuracy measured against reconciled outcomes. Conflating any two of these (the old ADR-117 framing that treated "agent memory" as one thing; the earlier ADR-194 v1 framing that treated Reviewer as an abstraction over the other three) produces confusion that ADR-189 and ADR-194 v2 respectively resolve.

## Product promise (canonical one-liner)

Every external-facing statement of what YARNNN is must be traceable to one of the two phrasings below. Other copy may riff on these for voice variation, but the substance must be preserved.

**Primary:** *Describe your work. Create the agents that do it.*

**Secondary (short form):** *Your work, your agents.*

These replace the prior canonical one-liner ("Persistent agents with accumulated context do recurring work products for you") in user-facing copy. The prior framing remains valid for internal/architectural contexts where the accumulation mechanism is the point.
