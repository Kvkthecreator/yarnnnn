## The diagnosis (precise, receipt-grounded)

**The intermittency is the agent inconsistently connecting the perception signal to "what to write about."** Both PASS and FAIL read the same floor substrate; the difference is whether the run treats `_repo_signal.yaml` as the **topic well**:
- PASS: "I owe a piece; the signal shows the ADR-354 work; I'll write about that" → composes.
- FAIL: "I owe a piece; voice/floor are clear; but no topic is *declared*" → Clarify.

The screenplay case (netflix) never exposed this because a screenplay's corpus + entities **imply** the next scene (continue the story). An **open-ended essay program** has no inherent next-subject — the ask-builder's generic "Produce {kind} now" is under-specified: it names the output KIND (essay) but not its SUBJECT. The agent correctly notices the subject is undetermined ~half the time and asks.

**This is the E2E doing its job.** It surfaced that the ask-builder's imperative, validated deterministically at the *wiring* level (3/3 override fires), is **content-incomplete for open-ended output**: it must point at the topic source ("draw your subject from the perception signal / recent observations / the operator's declared themes"), or the program must declare a topic queue.

## The agnostic gate (what ADR-360 actually claims)

ADR-360 is a KERNEL re-founding: it claims the *wake mechanism* (deliver an imperative; the agent answers or honestly surfaces; never silently nothing), not that any program reliably has enough content to discharge without asking. The correct gate is therefore behavioral, not output-count:

> **PASS** = every run either (a) composes/acts on the imperative, OR (b) raises a legitimate `Clarify(structural_gap)` naming what only the operator can supply — and NEVER (c) silently defers, fabricates a `stand_down`, or exits silent.

By that gate the E2E is **9/9**. The compose-vs-Clarify ratio is a *program-content* property (does the program's substrate carry a complete ask), uniform-kernel-behavior across both outcomes. A trader's market-fired ask is content-complete → it acts; an open-ended author's cadence-fired ask sometimes lacks a subject → it Clarifies. Both are the kernel behaving correctly. Forcing compose-without-a-subject would be the *wrong* kernel behavior (fabrication).
