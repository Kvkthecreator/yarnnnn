# Observability — Canonical Reference

**Version**: 1.0
**Established**: 2026-05-06 (ADR-250)
**Status**: Canonical — single authoritative reference for all observability, logging, and telemetry decisions.

> **Start here** before adding any logging, error tracking, alerting, or cost monitoring to YARNNN.
> ADR-250 records the architectural decision. This document is the living operational reference.

---

## Philosophy

YARNNN's observability model follows three principles:

**1. Substrate-first.** The primary observability record is Postgres (`execution_events` table), consistent with FOUNDATIONS Axiom 1. If you need to answer "what happened?", the answer is a SQL query, not a log grep.

**2. Sentry for exceptions, Postgres for analytics.** These are distinct concerns. Sentry captures unhandled exceptions with context, permanently. Postgres captures structured execution metrics for cost attribution, failure analysis, and trend reporting. Do not conflate them.

**3. Narrative is for operators, telemetry is for machines.** The narrative (chat surface) surfaces spend ceiling events and failure summaries in human-readable form. The `execution_events` table is the machine-readable counterpart. Never put analytics-critical data only in narrative prose.

---

## Stack

| Layer | Tool | Purpose | Cost |
|-------|------|---------|------|
| Error tracking | Sentry (free tier) | Unhandled exceptions, stack traces, performance traces | $0 (< 5K errors/month) |
| Execution ledger | `execution_events` (Postgres) | Every invocation: cost, tokens, status, error reason | $0 (existing DB) |
| Spend guard | In-process gate (dispatcher) | Daily spend ceiling before each generative invocation | $0 |
| Raw logs | Render stdout | Live debugging, transient tail during incidents | $0 (included in Render) |
| Billing ledger | `token_usage` (Postgres) | Monthly spend aggregation, user-facing billing | $0 (existing) |

**What is NOT in the stack (and why):**

- **Datadog / Grafana Cloud / Axiom** — designed for high-volume log streaming; overengineered for current scale; add when DAU > 1000 or log volume exceeds Render retention
- **PagerDuty / OpsGenie** — Sentry email alerts are sufficient; on-call rotation not warranted at alpha scale
- **Real-time metrics dashboards** — the admin page + `execution_events` SQL covers this; a separate metrics pipeline is redundant until multi-region or multi-cloud

## Layer 1: Sentry

### What it captures

- Every unhandled Python exception in `yarnnn-api` and `yarnnn-unified-scheduler`
- Every unhandled JS error in the Next.js web app
- Performance traces for invocations (start → tool rounds → complete)
- Contextual tags: `user_id`, `task_slug`, `shape`, `tool_rounds`, `environment`

### What it does NOT capture

- Expected failures (balance exhausted, capability unavailable) — these are business logic, not exceptions; they go to `execution_events` with `status='failed'`
- LLM API errors that are caught and handled — only unhandled exceptions propagate to Sentry

### Configuration

```python
# api/main.py and api/jobs/unified_scheduler.py
import sentry_sdk
sentry_sdk.init(
    dsn=os.getenv("SENTRY_DSN"),
    environment=os.getenv("ENVIRONMENT", "production"),
    traces_sample_rate=0.1,  # 10% of transactions traced (free tier headroom)
    send_default_pii=False,  # no PII in Sentry
)
```

```javascript
// web/sentry.client.config.ts (generated by @sentry/wizard)
Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  tracesSampleRate: 0.05,  // 5% frontend transactions
});
```

### Env vars (all 4 Render services)

| Var | Value | Services |
|-----|-------|---------|
| `SENTRY_DSN` | From sentry.io project settings | API, Scheduler, Web |
| `ENVIRONMENT` | `production` | API, Scheduler |

### Alerting policy

Sentry default: email on first occurrence of a new error class. Do not configure alert fatigue-inducing rules. The two alerts worth adding manually in the Sentry UI:

1. **Anthropic API errors** — filter: `anthropic.BadRequestError` OR `anthropic.RateLimitError`. Action: email immediately.
2. **Dispatch failure rate** — filter: event_type contains `[DISPATCH]` AND level=ERROR, threshold: >5 in 1 hour. Action: email.

## Layer 2: `execution_events` Table

### Purpose

The authoritative structured record of every invocation attempt. One row per invocation, always written, regardless of outcome. This is what answers every "what happened and what did it cost?" question.

### Schema

```sql
CREATE TABLE execution_events (
    id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             uuid NOT NULL REFERENCES auth.users(id),
    slug                text NOT NULL,
    shape               text NOT NULL,       -- deliverable | accumulation | action | maintenance
    trigger_type        text NOT NULL,       -- scheduled | manual | back_office
    status              text NOT NULL,       -- success | failed | skipped
    error_reason        text,               -- see Error Reason Taxonomy below
    error_detail        text,               -- exception message, max 2000 chars
    tool_rounds         int,
    input_tokens        bigint,
    output_tokens       bigint,
    cache_read_tokens   bigint,
    cache_create_tokens bigint,
    cost_usd            numeric(10,6),      -- cache-inclusive (see Cost Formula)
    duration_ms         int,
    agent_run_id        uuid REFERENCES agent_runs(id),  -- NULL for failures
    created_at          timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_execution_events_user_date ON execution_events (user_id, created_at DESC);
CREATE INDEX idx_execution_events_slug      ON execution_events (user_id, slug, created_at DESC);
CREATE INDEX idx_execution_events_status    ON execution_events (status, created_at DESC);
```

### Error Reason Taxonomy

| `error_reason` | Meaning | Written by |
|----------------|---------|-----------|
| `balance_exhausted` | Anthropic API returned credit balance too low | dispatcher early-exit |
| `capability_unavailable` | Required platform capability not connected | dispatcher early-exit |
| `spend_ceiling` | Daily spend limit reached before invocation started | spend guard |
| `exception` | Unhandled Python exception during dispatch | dispatcher outer except |
| `timeout` | Invocation exceeded max duration (future) | watchdog |
| `empty_draft` | LLM returned no usable content | dispatcher generation path |
| NULL | Success or skipped-without-error | — |

### Cost Formula

`execution_events.cost_usd` uses the **cache-inclusive formula**, which is the accurate Anthropic billing model:

```python
def compute_cost_usd_inclusive(
    model: str,
    input_tokens: int,
    output_tokens: int,
    cache_read_tokens: int = 0,
    cache_create_tokens: int = 0,
) -> float:
    rate = BILLING_RATES.get(model, _DEFAULT_BILLING_RATE)
    return round(
        (input_tokens       / 1_000_000) * rate["input"]
      + (cache_read_tokens  / 1_000_000) * rate["input"] * 0.10
      + (cache_create_tokens/ 1_000_000) * rate["input"] * 1.25
      + (output_tokens      / 1_000_000) * rate["output"],
        6
    )
```

**Note:** `token_usage.cost_usd` uses the older cache-agnostic formula and underreports by ~15–20% on high-cache-hit runs. It is retained as the billing ledger (user-facing). `execution_events.cost_usd` is the accurate telemetry figure. Do not mix them.

### Write discipline

The single write path is `api/services/telemetry.py::record_execution_event(...)`. Rules:

1. Called exactly once per invocation attempt, at completion (success or early exit)
2. Never raises — wrapped in try/except, logs on failure, does not block the invocation
3. `agent_run_id` is populated only when an `agent_runs` row was created
4. `error_detail` truncated to 2000 chars to prevent unbounded storage

### Key queries

```sql
-- Per-task cost this week
SELECT slug, COUNT(*) as runs, SUM(cost_usd) as total_cost_usd,
       SUM(CASE WHEN status='failed' THEN 1 ELSE 0 END) as failures
FROM execution_events
WHERE user_id = $1 AND created_at >= NOW() - INTERVAL '7 days'
GROUP BY slug ORDER BY total_cost_usd DESC;

-- Today's spend (for spend guard)
SELECT COALESCE(SUM(cost_usd), 0)
FROM execution_events
WHERE user_id = $1 AND created_at >= DATE_TRUNC('day', NOW() AT TIME ZONE 'UTC');

-- Runaway detection: runs with >8 tool rounds
SELECT slug, tool_rounds, cost_usd, created_at
FROM execution_events
WHERE user_id = $1 AND tool_rounds > 8
ORDER BY created_at DESC LIMIT 20;

-- Failure breakdown
SELECT error_reason, COUNT(*) as count
FROM execution_events
WHERE user_id = $1 AND status = 'failed'
  AND created_at >= NOW() - INTERVAL '7 days'
GROUP BY error_reason ORDER BY count DESC;
```

## Layer 3: Daily Spend Guard

### Purpose

Circuit breaker at the dispatcher level. Prevents runaway costs by checking cumulative daily spend before starting any generative invocation.

### Location

`api/services/invocation_dispatcher.py` — checked after capability validation, before generation starts.

### Configuration

| Env var | Default | Description |
|---------|---------|-------------|
| `DAILY_SPEND_CEILING_USD` | `10.0` | Per-user daily spend ceiling. Block generative invocations above this. |

Set on: `yarnnn-api` + `yarnnn-unified-scheduler`.

### Behavior

```
Scheduled invocation arrives
  → check capability (existing)
  → check daily spend from execution_events        ← NEW
      if spend >= ceiling AND trigger = scheduled:
          write execution_events row (status=skipped, error_reason=spend_ceiling)
          emit narrative entry (spend summary, weight=routine)
          return early
      if spend >= ceiling AND trigger = manual:
          emit narrative warning (not a block — user explicitly requested)
          continue
  → generate (existing)
```

**Maintenance shape is always exempt** — back-office tasks make no LLM calls; blocking them on spend ceiling would break system hygiene.

### Narrative entry format (when ceiling hit)

```
role: system
weight: routine
summary: "Spend ceiling reached — {slug} skipped"
body: "{slug} was due at {time} but today's spend (${X.XX}) has reached the
       daily ceiling (${Y.00}). The run was skipped. Reset at midnight UTC.
       Adjust DAILY_SPEND_CEILING_USD to change the limit."
```

## Layer 4: Render Stdout (Raw Logs)

### What it is

Every `logger.info()`, `logger.warning()`, `logger.error()` call in the Python backend goes to Render's log stream. Accessible via Render dashboard or Render MCP `list_logs`.

### Retention

~7 days. Do not rely on stdout for anything that needs to persist longer. That is what `execution_events` is for.

### Log format conventions

Existing conventions (do not change):

```
INFO:__main__:[SCHED] ✓ {user_id[:8]}/{slug}: {result_message}
INFO:__main__:[SCHED] ✗ {user_id[:8]}/{slug}: {error_message}
ERROR:services.invocation_dispatcher:[DISPATCH] {shape}/{slug} failed: {error}
INFO:services.invocation_dispatcher:[DISPATCH] {shape}/{slug} complete v{n} ({ms}ms)
WARNING:services.back_office.reviewer_calibration:[REVIEWER_CALIBRATION] rebuild failed: {error}
```

When adding new log lines, follow the `[MODULE_PREFIX]` convention so log filters work.

### When to use vs. Sentry vs. execution_events

| Situation | Use |
|-----------|-----|
| Debugging a live incident right now | Render stdout (tail in Render dashboard) |
| Finding why a specific run failed (historical) | `execution_events.error_detail` + Sentry |
| Tracking cost trends | `execution_events` SQL |
| Getting alerted on a new exception class | Sentry |
| Checking if scheduler is alive | `activity_log` scheduler_heartbeat |

## Layer 5: `token_usage` Table (Billing Ledger)

The `token_usage` table predates ADR-250 and remains as the **billing ledger** — it drives user-facing balance calculations and monthly spend aggregation in the admin dashboard.

**Do not remove or migrate away from `token_usage`.** It is the billing source of truth.

**Do not use `token_usage.cost_usd` for telemetry analytics** — it is cache-agnostic and underreports by ~15–20%. Use `execution_events.cost_usd` for all analytical queries.

The two tables serve different masters:

| Table | Owner | Purpose | Cost formula |
|-------|-------|---------|-------------|
| `token_usage` | Billing system | User balance, monthly spend | Cache-agnostic (legacy) |
| `execution_events` | Telemetry system | Per-run analytics, cost attribution | Cache-inclusive (accurate) |

---

## `activity_log` — What It Is and Is Not

`activity_log` is the **workspace lifecycle event log** — it records high-level operator-visible state changes (integration connected, agent approved, scheduler heartbeat). It is NOT an execution telemetry log.

**Write to `activity_log` for:** integration_connected, agent_approved, agent_rejected, chat_session, scheduler_heartbeat.

**Do NOT write to `activity_log` for:** individual invocation outcomes, token counts, per-run errors. That is `execution_events`.

**Known issue:** `scheduler_heartbeat` is written ~6× per minute due to a duplicate-write bug in `unified_scheduler.py`. This is noise and should be deduplicated (write once per cycle, not once per user per cycle).

## Admin Dashboard Surface

The admin dashboard at `/admin` reads from:

- `execution_events` — per-task cost, failure counts, run history (post ADR-250)
- `token_usage` — monthly spend (billing-accurate)
- `agent_runs` — legacy run counts and avg token display
- `activity_log` — scheduler heartbeat recency

Post ADR-250 implementation, the per-task breakdown table gains:
- `cost_usd` (sum from `execution_events`, cache-inclusive)
- `failed_count` (from `execution_events` WHERE status='failed')
- `skipped_count` (from `execution_events` WHERE status='skipped')

The daily spend rate card shows today's spend vs. `DAILY_SPEND_CEILING_USD`.

## Runaway Detection

A "runaway" invocation is one that consumes disproportionate tokens relative to its purpose. Indicators:

| Signal | Threshold | Source |
|--------|-----------|--------|
| Tool rounds | > 8 in one run | `execution_events.tool_rounds` |
| Input tokens | > 300K in one run | `execution_events.input_tokens` |
| Cost per run | > $2.00 | `execution_events.cost_usd` |
| Daily spend | > `DAILY_SPEND_CEILING_USD` | spend guard (blocks automatically) |

The spend guard (Layer 3) is the active circuit breaker. The `execution_events` table is the passive record for post-hoc detection.

For the May 2026 incident specifically: `track-universe-2` hit 12 tool rounds with 670K–755K input tokens per run ($3–4 per run). With a `DAILY_SPEND_CEILING_USD=10.0`, the second run of the day (4am PT) would have been blocked, saving ~$8 of the $4.80 charged on May 4th.

## Future Considerations

These are explicitly out of scope for ADR-250 and should be addressed in subsequent ADRs when pressure justifies:

- **Per-tool cost breakdown** — record cost at the tool-call level, not just per-invocation. Requires changes to dispatch_helpers.py tool loop.
- **Real-time spend meter** — surface current-run token accumulation in the cockpit during execution. Requires a websocket or polling pattern.
- **Per-user configurable ceiling** — allow operators to set `spend_ceiling_usd` in AUTONOMY.md. Currently a single env var for the whole platform.
- **External log aggregation** — Axiom, Grafana Loki, or Datadog when DAU > 1000 or Render's 7-day log retention becomes a liability.
- **Structured tracing (OpenTelemetry)** — OTel spans across scheduler → dispatcher → generation would enable deep latency profiling. Worth adding when invocation p95 latency becomes a user-visible concern.
- **Sentry Performance** — currently set to 10% sample rate (free tier). Increase when per-invocation latency breakdown is needed for optimization.
