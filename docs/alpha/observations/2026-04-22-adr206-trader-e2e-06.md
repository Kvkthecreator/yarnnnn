# Observation 06 — YARNNN delegated Intent-file authorship via task trigger_context (architectural question)

**Date**: 2026-04-22
**E2E run**: post-purge ADR-206 trader validation
**Persona**: alpha-trader
**Phase**: after second scaffold turn
**Severity**: **HIGH — architectural question, not a bug but a decision point**

## What happened

I explicitly asked YARNNN to write three files: `_operator_profile.md`
and `_risk.md` at `/workspace/context/trading/`, and to overwrite
`/workspace/review/principles.md` with the declared capital-EV framework.

YARNNN's strategy was not to write these files directly. Instead, YARNNN:

1. Put the full signal library, sizing formula, portfolio limits, and
   proposal rules into `/agents/trading-operator/AGENT.md` via `EditEntity`
   (updating the deprecated `agent_instructions` column).
2. Triggered `track-universe` with a bootstrap `trigger_context.md`
   containing explicit instructions for the trading-operator analyst
   agent to write the three Intent-layer files on its first run:

   > "BOOTSTRAP RUN. Before collecting market data, write three
   > Intent-layer files: (1) /workspace/context/trading/_operator_profile.md
   > ... (2) /workspace/context/trading/_risk.md ... (3) /workspace/review/
   > principles.md ... After writing those three files, proceed with the
   > normal market data collection..."

**The agent_run for track-universe is stuck in `generating` status with
0 bytes of draft content** — either hung, or silently failed. Neither
the domain Intent files nor the customized principles.md were written.

## The architectural question

Should Intent-layer files be authored by:

- **(a) The operator directly** (typed into a modal / chat), or
- **(b) YARNNN on the operator's behalf** (YARNNN calls WriteFile or a
  dedicated primitive), or
- **(c) A domain agent via task bootstrap** (YARNNN delegates the
  authorship to an agent's first run via trigger_context)?

YARNNN chose (c). This is interesting because:

- **Pro**: The domain agent has best vocabulary-fit for its domain.
  A trading operator agent "knows" what a good `_operator_profile.md`
  for trading looks like.
- **Pro**: The authorship happens via the task pipeline, which gets
  proper RuntimeDispatch, prompt calibration, token accounting.
- **Pro**: Separation of concerns — YARNNN orchestrates, agents execute.
- **Con**: The Reviewer's `principles.md` in particular is supposed to be
  the *operator's declared framework*, not the domain agent's inferred
  framework. Delegating it to the trading-operator agent inverts the
  authorship chain. The operator declared the framework explicitly —
  YARNNN should have honored that directly.
- **Con**: If the first task run fails (as it did — stuck in `generating`),
  the Intent artifacts never get authored, and the operator doesn't know
  why their stated framework isn't reflected in substrate.

## Proposed resolution

**Split by artifact.** Intent files have different authorship contracts:

| File | Authored by | Rationale |
|---|---|---|
| `/workspace/context/_shared/IDENTITY.md` | YARNNN via UpdateContext inference | Workspace-level; YARNNN has context |
| `/workspace/context/_shared/BRAND.md` | YARNNN via UpdateContext inference | Same as above |
| `/workspace/context/{domain}/_operator_profile.md` | YARNNN or domain agent | Either is OK; probably YARNNN for first write, agent can refine |
| `/workspace/context/{domain}/_risk.md` | **Operator authors directly** | Hard limits. Must not be inferred — operator-declared. |
| `/workspace/review/principles.md` | **Operator authors directly** | Reviewer framework. Calibrates capital-EV gates. Operator-owned. |

The current primitive surface doesn't cleanly support this split. Fix
candidates:

1. Add `UpdateContext(target="risk", domain="trading")` and
   `UpdateContext(target="review_principles")` — inference-capable but
   strict about preserving operator wording verbatim when provided.
2. Add explicit modal affordances for `/workspace/context/{domain}/_risk.md`
   and `/workspace/review/principles.md` in `ManageContextModal` (currently
   only handles workspace-level IDENTITY/BRAND/CONVENTIONS).
3. Tighten YARNNN prompt to prefer direct authorship of these two files
   over delegation-via-trigger-context. "If the operator declares a risk
   framework or Reviewer framework explicitly, write it to the file
   verbatim; do not delegate to a task."

## What this run teaches

The E2E exposed a **real unresolved decision in the ADR-206 architecture**:
who authors Intent-layer files, in what cases, and via what primitive.
ADR-206 specifies the files but doesn't specify the authorship chain.
YARNNN improvised — imperfectly.

This is ADR-207 material.

**Classification**: A-system. Unresolved architectural decision.
