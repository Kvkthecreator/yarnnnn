# Observation 07 — Task pipeline silently hangs on first real dispatch

**Date**: 2026-04-22
**E2E run**: post-purge ADR-206 trader validation
**Persona**: alpha-trader
**Phase**: first task trigger (track-universe via ManageTask action="trigger")
**Severity**: **CRITICAL — the loop cannot close**

## What happened

After creating the four tasks, YARNNN called:

```
ManageTask(
  action="trigger",
  task_slug="track-universe",
  context="BOOTSTRAP RUN. Before collecting market data, write three
           Intent-layer files: (1) /workspace/context/trading/_operator_profile.md
           (2) /workspace/context/trading/_risk.md
           (3) /workspace/review/principles.md
           After writing those three files, proceed with the normal market
           data collection for all 15 universe instruments."
)
```

This created `agent_runs` row `de13d82f-4a94-48ed-b1ee-7fbbfbb101de` with
`status="generating"` at 05:51:22 UTC.

**5.5 minutes later** (as of this observation):
- `status` still `generating`
- `draft_content` = 0 bytes
- `final_content` = 0 bytes
- No activity_log entry for this run
- No writes to `/workspace/context/trading/*` or `/tasks/track-universe/outputs/*`

The pipeline initiated the run, created the DB row, wrote
`trigger_context.md`, and then... silently failed. No error status. No
completion status. No logged exception.

## Why this is the most important E2E finding

Every other observation (01-06) is fixable with scoped commits. This one
means **the operation loop cannot close**. Without a completed task run,
there's no output, no proposal, no Reviewer evaluation, no Queue entry,
no approval, no execution, no money-truth. The entire ADR-206 loop —
Intent → Operation → Deliverables — dies at the first arrow.

If this reproduces, alpha-trader's first session goes:
1. Operator declares operation ✓
2. YARNNN scaffolds IDENTITY/BRAND/Agent/Tasks ✓
3. YARNNN triggers first task ✗ — hangs silently
4. Operator waits; nothing arrives
5. Operator loses trust in the system

## Hypotheses for root cause

### Hypothesis A: Specialist lazy-ensure gap in task_pipeline.py

The `track-universe` TASK.md declares `team: [researcher, tracker]` but
the agent_slug is `trading-operator` (role=analyst). The pipeline may
try to resolve researcher or tracker specialists for the process steps
and fail — ADR-205's `ensure_infrastructure_agent` should handle this
but may have an unhandled path for role-resolution from TASK.md team
sections vs. agent_slug column.

**Test**: Check `task_pipeline.py` step resolution — does it honor the
`## Team` section in TASK.md or only the `agent_slug` column on the
`tasks` row? If it tries to dispatch to `researcher` or `tracker`
directly and `ensure_infrastructure_agent` doesn't fire for these via
the trigger path (only via normal dispatch path), we get a silent hang.

### Hypothesis B: LLM call upstream error not propagated

The Sonnet call for the agent's generation phase may have errored
(auth, rate limit, timeout, API error) without the handler updating
the `agent_runs.status`. Worth checking by:
- Inspecting Render logs for the yarnnn-api service around 05:51:22 UTC
- Checking if `token_usage` has a corresponding entry (would indicate
  the call at least started)

### Hypothesis C: `trigger_context.md` too large for the generation prompt

The bootstrap context I observed YARNNN write is ~600 characters of
multi-line markdown with explicit instructions. Possible that the
task pipeline's prompt assembly can't handle large trigger_context
contents and silently truncates or errors.

### Hypothesis D: The pipeline tried to dispatch and found no `_operator_profile.md` / `_risk.md` (they don't exist yet), then crashed on missing context

`gather_task_context()` in `task_pipeline.py` reads domain `_operator_profile.md`
if `context_reads` in TASK.md includes `trading`. If the read fails and
there's no fallback, the pipeline could crash pre-generation.

and a focus on pipeline robustness before the next E2E.
