# Finding — ADR-292 Bundle-Update Gate Cannot Distinguish "Old Bundle Content" from "Operator-Customized Content"

*Surfaced 2026-05-20 during yarnnn-author autonomy-demonstration setup.*

## Headline

ADR-292's `apply_substrate_update(scope="bundle")` reuses `fork_reference_workspace`'s `is_skeleton_content` gate to decide whether to push bundle updates into a live workspace. The gate compares existing-workspace-content against the bundle's current copy: if they match (skeleton), update; if they differ (assumed customized), skip.

This gate **cannot distinguish** two structurally different cases:
1. Operator-customized content (the case the gate is designed to protect — don't bulldoze operator work)
2. **Stale bundle content** — workspace has the bundle's version from N days ago; bundle has since been updated; the workspace's content differs from the bundle's current copy *only because the bundle was updated*

The gate skips both cases identically, but case 2 is precisely what ADR-292 is meant to push. The result: bundle updates that haven't ever been operator-touched don't propagate to live workspaces.

## How this surfaced

Setting up yarnnn-author for the autonomy demonstration:
1. ADR-295 Phase A (commit `bd07e5e`) hardened `docs/programs/alpha-author/reference-workspace/review/principles.md` with the Self-Improvement Posture section (~7K new content; total ~18K chars).
2. Activation of yarnnn-author had previously forked the older bundle principles.md into the workspace on 2026-05-18 (~11K chars).
3. Today's `activate_persona.py` and `apply_substrate_update(scope="bundle")` BOTH skipped `review/principles.md` — same gate, same skip-outcome.
4. Verified via `workspace_file_versions`: all 4 prior revisions of principles.md in this workspace are `authored_by: system:bundle-fork` — there are zero operator or Reviewer revisions. The content is *purely* a stale bundle fork.
5. Yet the gate classified it as "operator-customized" because workspace content ≠ bundle content.

## Workaround applied

Direct write via `write_revision` with `authored_by: system:substrate-update` (the canonical ADR-292 attribution actor) — script at `api/scripts/alpha_ops/_force_push_principles.py`. This bypasses the gate by writing directly. **Workaround is developer-side scaffolding, not a system fix.** Real operators would not have this script available.

## What this means in production

For any bundle file that has been previously bundle-forked into a live workspace and subsequently updated in the bundle:
- The operator's cockpit Settings → Workspace "Update bundle" button (ADR-292 Phase 2, when shipped) will NOT push the updated content
- The operator's workspace will silently run the older bundle version indefinitely
- The operator has no surface signal that an update was available + skipped

This is a **silent drift** between bundle canon and live workspaces. It defeats ADR-292's purpose.

## Why `is_skeleton_content` made sense initially (and where it breaks)

The gate works correctly for the **initial activation** case:
- Workspace has no file → write it (case: content is None)
- Workspace has file matching bundle skeleton → re-write (case: content == bundle's skeleton/template)
- Workspace has file not matching bundle → skip (case: operator customized)

The third branch's assumption "if content ≠ bundle, operator customized" was correct under the **assumption that the bundle never updates between activations**. The moment the bundle changes (which is the entire premise of ADR-292), the assumption breaks.

## Recommended Hat-A amendments

Three possible system-canon fixes, in increasing order of architectural commitment:

**Option 1 — Version-tracking gate.**
Track each bundle file's version (content hash or bundle MANIFEST version) at fork-time. Store as workspace_files metadata or in a sibling table. When `apply_substrate_update` runs, the gate becomes:
- Workspace has file → check its bundle-version-stamp
- If stamp matches current bundle version → no-op (workspace has latest)
- If stamp differs AND no operator-revision on top → **push the update** (this is the case Option 1 captures that current gate misses)
- If stamp differs AND operator-revision present → skip (operator customized after fork)

Distinguishes "stale bundle content" from "operator-customized content" cleanly. Requires schema addition (`bundle_version_at_fork` per file).

**Option 2 — Revision-chain-aware gate.**
Read the file's revision chain. If ALL prior revisions are `system:bundle-fork` (no operator / reviewer revisions), classify as "stale bundle content" → push. If any non-system-bundle-fork revision exists, classify as "customized" → skip.

This is **code-only**, no schema change. Cleaner. But it relies on attribution discipline holding (e.g., if a Reviewer wrote to principles.md, its attribution is `reviewer:*` and the gate correctly skips; if an operator edited via Files surface, attribution is `operator:*` and gate skips). The discipline is already in place per ADR-209.

**Recommended option.** This is what ADR-292 v2 should ship.

**Option 3 — Per-file force-update flag.**
Add a `--force-paths {path1,path2,...}` parameter to `apply_substrate_update` that bypasses the gate for specific paths. Lets the operator (via cockpit Phase 2 UI) say "I know I want this update; push it." Combines well with Option 2.
