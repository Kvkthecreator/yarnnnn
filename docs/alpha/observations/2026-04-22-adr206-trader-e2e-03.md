# Observation 03 — domain-scoped Intent artifacts have no primitive write target

**Date**: 2026-04-22
**E2E run**: post-purge ADR-206 trader validation
**Persona**: alpha-trader
**Phase**: first scaffold turn (operator declaration → YARNNN scaffold)
**Severity**: **HIGH — real architecture gap surfaced by the E2E**

## What happened

I declared the full Simons-inspired trading operation in a single chat turn
and asked YARNNN to "author the operator profile, risk file, Reviewer
principles at the trading paths, and scaffold the tasks."

YARNNN:
- Wrote `/workspace/context/_shared/IDENTITY.md` (Simons-consistent) ✓
- Wrote `/workspace/context/_shared/BRAND.md` (inferred) ✓
- Called `ManageAgent(action="create", title="Trading Operator", role="analyst")`
  and wrote `/agents/trading-operator/AGENT.md` containing the full signal
  library, sizing math, risk caps, portfolio limits, and execution rules ✓

YARNNN did NOT:
- Write `/workspace/context/trading/_operator_profile.md` ✗
- Write `/workspace/context/trading/_risk.md` ✗
- Customize `/workspace/review/principles.md` with the declared capital-EV
  check ladder ✗

## The root cause

ADR-206 §"The three layers" specifies:

> **Operation purpose** — `/workspace/context/{domain}/_operator_profile.md`
> + `/workspace/context/{domain}/_risk.md`

But the primitive surface (per `docs/architecture/primitives-matrix.md`) has
no target for these files. `UpdateContext` accepts
`target in ("workspace", "identity", "brand", "memory", "agent", "task",
"awareness")`. There is no `target="operator_profile"` or `target="risk"`,
and no path-scoped variant like `target="domain", domain="trading",
file="_operator_profile"`.

YARNNN fell back to the next-most-appropriate surface it did have — the
`/agents/trading-operator/AGENT.md` — and put the entire operation spec
there. That's not wrong (Agent-authored-context per ADR-189), but it's
also not what ADR-206 claims the substrate should look like.

## Two honest resolutions

### (a) Add primitive targets for domain-Intent files

Extend `UpdateContext`:
```
target="operator_profile"  →  /workspace/context/{domain}/_operator_profile.md
target="risk"              →  /workspace/context/{domain}/_risk.md
target="review_principles" →  /workspace/review/principles.md
```
These are first-class ADR-206 Intent artifacts. Making them writable via
`UpdateContext` (with inference support) is the most consistent path.

### (b) Retract ADR-206's domain-Intent claims; AGENT.md is the Intent home

If we accept that `_operator_profile.md` is redundant with a well-authored
AGENT.md for the operation's primary Agent (in this case `trading-operator`),
then ADR-206 overspecified. The "authored Agent" IS the operator profile;
no separate file needed. Review principles remains at `/workspace/review/`
but is cumstomized via chat + generic `WriteFile` primitive.

## My recommendation

**Resolution (a).** ADR-206's commitment to the three-layer operator view
is load-bearing — Intent is first-class, and Intent artifacts need to be
writable via a first-class primitive. Falling back to AGENT.md collapses
Intent into Operation, which is exactly what ADR-206 rejected.

Specifically:
- Extend `UpdateContext` with `target="operator_profile"` and `target="risk"`
  that resolve to the current domain. Since domain isn't always obvious,
  add a required `domain` parameter when target is one of these.
- Extend `UpdateContext` with `target="review_principles"` for the workspace
  Reviewer principles file.
- Update the YARNNN prompt to prefer these targets for Simons/Commerce-style
  declared operations.

## Classification

**A (system/architecture).** Real gap in primitive surface vs. ADR claim.
ADR seed.

## Did it block the E2E?

No. YARNNN successfully wrote the operator spec to AGENT.md, which is
functionally close enough for the rest of the E2E to proceed. But the ADR-206
substrate shape I promised isn't actually produced by YARNNN's current
tool surface. Fix in a follow-up commit.
