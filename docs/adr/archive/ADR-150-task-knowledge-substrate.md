# ADR-150: Task Knowledge Substrate — Accumulation as Primary Artifact

**Status:** Superseded by ADR-151 (shared knowledge domains). Task-gated knowledge was architecturally wrong — it contradicts Axiom 2 (shared workspace as OS) and prevents cross-task knowledge compounding. The core insight (knowledge as primary artifact, output as derivative) was correct and is preserved in ADR-151. The error was scoping knowledge to tasks instead of the shared workspace.  
**Date:** 2026-03-31  
**Extends:** ADR-149 (task lifecycle), ADR-138 (agents as work units), ADR-141 (unified execution)  
**Implements:** FOUNDATIONS.md Axiom 2 (recursive perception), Axiom 3 (agents as developing entities), Axiom 4 (value from accumulated attention)

---

## Context

ADR-149 established DELIVERABLE.md as the quality contract and mode-aware output semantics. But the execution model remains stateless — each task run gathers fresh context, generates a report, and delivers it. The only continuity between runs is feedback.md and steering.md.

This contradicts the canonical axioms:

- **Axiom 2**: "Optimize for Accumulation, Not Extraction. The enduring value is in the recursive accumulation."
- **Axiom 3**: "A tenured agent produces better output because it knows more about its domain, not because it has more tools."
- **Axiom 4**: "A Slack digest is valuable because it summarizes today *knowing what it summarized yesterday*."
- **The Information Hierarchy**: Agents should ascend from L1 (digests) to L3 (analysis) through accumulated L4 (user knowledge). This requires persistent, structured knowledge — not just output history.

The current architecture treats each run as independent. A competitive intel agent on run 10 knows no more about Acme Corp than it did on run 1 — it re-researches from scratch every cycle. The accumulated knowledge that should make run 10 dramatically better than run 1 has nowhere to live.

## Decision

### Core Thesis: Knowledge Is the Primary Artifact, Output Is the Derivative

Every task maintains a **knowledge substrate** — structured, accumulated intelligence files that deepen with each execution cycle. The deliverable (report, brief, digest) is a **derivative** — a view over the accumulated knowledge, not the primary artifact.

This inverts the execution model:

| | Before (ADR-141/149) | After (ADR-150) |
|---|---|---|
| **Primary artifact** | The output (report) | The knowledge substrate (per-entity files) |
| **What persists** | Output archive | Structured knowledge that deepens over time |
| **What gets delivered** | The full report | A derivative — informed by accumulated knowledge |
| **Agent context for next run** | Platform content + web search | Its own accumulated knowledge + new signals |
| **What compounds** | Feedback preferences | The knowledge itself |

This applies to ALL task modes:

- **Recurring**: Knowledge files grow each cycle. Output = "what changed" digest over accumulated knowledge.
- **Goal**: Knowledge files fill in toward completion criteria. Output = comprehensive synthesis of accumulated findings.
- **Reactive**: Knowledge files persist across triggers. Each trigger benefits from all prior accumulated knowledge for this task.

### Filesystem Hierarchy — Three Scopes, Strict Boundaries

Knowledge accumulation creates a tension: where do files live, and who can see them? Three scopes, each with clear ownership:

**Workspace scope** (`/workspace/`) — USER-contributed, globally visible:
```
/workspace/
├── IDENTITY.md                    # User identity
├── BRAND.md                       # Output style/tone
├── documents/                     # User-uploaded reference material
│   ├── company-logo.png           # Workspace-wide asset, any task can reference
│   └── product-roadmap.md
└── assets/                        # User-uploaded shared assets
    └── logo-dark.svg              # (not generated by tasks — user-contributed only)
```

**Agent scope** (`/agents/{slug}/`) — METHODOLOGY, cross-task:
```
/agents/{slug}/
├── AGENT.md                       # Identity + instructions
├── memory/
│   ├── reflections.md             # Self-reflection (ADR-149)
│   ├── feedback.md                # Cross-task behavioral feedback
│   └── playbook-*.md              # Craft knowledge
└── (no domain knowledge — agents carry skill, not subject expertise)
```

**Task scope** (`/tasks/{slug}/`) — DOMAIN KNOWLEDGE + derived outputs:
```
/tasks/{slug}/
├── TASK.md                        # Charter: objective, process, mode
├── DELIVERABLE.md                 # Output specification (what the derivative looks like)
├── knowledge/                     # ACCUMULATED INTELLIGENCE (primary artifact)
│   ├── {entity-folder}/           # Per-entity knowledge (type-dependent)
│   │   ├── {entity}.md            # e.g., competitors/acme-corp.md
│   │   └── {entity}.md
│   ├── landscape.md               # Cross-entity synthesis (optional)
│   └── signals/                   # Temporal signal log (optional)
│       └── {date}.md
├── assets/                        # PERSISTENT ASSETS (updated with knowledge changes)
│   ├── competitor-matrix.svg      # Evolves as competitor knowledge deepens
│   ├── market-map.svg             # Updated when landscape.md changes
│   └── manifest.json              # Asset inventory + render metadata
├── outputs/                       # DERIVED DELIVERABLES (views over knowledge)
│   ├── latest/
│   │   ├── output.md
│   │   ├── output.html
│   │   └── manifest.json          # References task assets + per-run assets
│   └── {date}/
├── memory/
│   ├── run_log.md
│   ├── feedback.md                # User corrections + TP evaluations (ADR-149)
│   └── steering.md                # TP management notes (ADR-149)
└── working/                       # Ephemeral scratch (24h TTL)
```

### Three File Types, Three Lifecycles

| Type | Location | Lifecycle | Created by | Referenced by |
|---|---|---|---|---|
| **Workspace assets** | `/workspace/documents/`, `/workspace/assets/` | Permanent (user-uploaded) | User | Any task, any agent, TP |
| **Task knowledge** | `/tasks/{slug}/knowledge/` | Accumulating (grows each run) | Agent during execution | This task's agents. TP for cross-task. |
| **Task assets** | `/tasks/{slug}/assets/` | Persistent (updated with knowledge) | Agent / render pipeline | This task's outputs. DELIVERABLE.md refs. |
| **Output artifacts** | `/tasks/{slug}/outputs/{date}/` | Archival (one set per run) | Pipeline post-generation | Delivery, user review |

**The critical distinction: task assets vs. output artifacts.**

Task assets are **persistent visualizations of accumulated knowledge** — a competitive positioning chart that evolves as competitor data deepens, a relationship map that grows as contact knowledge accumulates. They live at `/tasks/{slug}/assets/` and get updated when the underlying knowledge changes.

Output artifacts are **per-run snapshots** — trend charts from this cycle's data, tables backing this week's analysis. They live inside `outputs/{date}/` and are bundled with the delivered output.

The derived output (the report/brief/digest) **references** persistent task assets rather than regenerating them:

```markdown
<!-- In output.md: -->
## Competitive Positioning
![Competitive positioning](../assets/competitor-matrix.svg)
*Updated 2026-03-31 — Acme Corp moved to enterprise tier after Series C*
```

### Cross-Task Knowledge Visibility

Task knowledge is **task-scoped by default**. Cross-task intelligence sharing is TP's job:

- TP reads from one task's knowledge, injects relevant context into another task's steering notes
- TP can copy findings between tasks when orchestrating (e.g., "competitive intel found Acme raised $50M — add to stakeholder update steering")
- No automatic cross-task filesystem symlinks — TP's judgment determines relevance

This preserves clean boundaries while allowing orchestrated knowledge flow. The agent writing competitive intel doesn't need to know about the stakeholder update task — TP bridges that gap.

### Context Referencing Priority

When an agent executes a task, context is loaded in priority order:

1. **DELIVERABLE.md** — what to produce (quality contract)
2. **knowledge/** — accumulated domain intelligence (PRIMARY CONTEXT)
3. **assets/** — persistent visualizations to reference/update
4. **memory/steering.md** — TP's cycle-specific guidance
5. **memory/feedback.md** — recent corrections
6. `/workspace/IDENTITY.md` + `BRAND.md` — user identity for tone/audience
7. `/workspace/documents/` — user-uploaded references (if relevant)
8. **Platform content** — web search, Slack, Notion (for NEW signals only)

The hierarchy is intentional: **accumulated knowledge comes before fresh signals**. The agent reasons from what it already knows, then researches what's new.

### DELIVERABLE.md Asset Specification (Extended)

DELIVERABLE.md gains asset categorization:

```markdown
## Expected Assets

### Persistent (task-level, updated with knowledge)
- Competitive positioning chart: quadrant diagram, updated when competitor assessments change
- Market landscape map: mermaid diagram showing player positions and movements

### Per-Output (generated fresh each cycle)
- Trend charts: quantified data from this cycle's signals
- Data tables: raw data backing this cycle's analysis

```

### Knowledge Schemas by Task Type Category

| Category | Entity type | Knowledge structure | Derivative output |
|---|---|---|---|
| **Intelligence** (competitive-intel, market-research) | Competitors, market segments | Per-entity files + landscape synthesis | What changed + implications |
| **Operations** (stakeholder-update, project-status) | Workstreams, milestones | Per-workstream files + status synthesis | Progress delta + blockers |
| **Platform** (slack-recap, notion-sync) | Channels, pages | Per-source signal accumulation | Activity digest + patterns |
| **Content** (content-brief, launch-material) | Topics, audiences | Per-topic research + drafts | Assembled deliverable from research |
| **Tracking** (gtm-tracker, relationship-health) | Contacts, channels | Per-entity tracking files | Health/performance dashboard |

Not all types need entity-level knowledge. Simpler types (slack-recap) may just have a signal log. Complex types (competitive-intel) have full per-entity tracking. The schema is declared per type, not forced on all.

### Pipeline Changes

The execution model gains a knowledge update phase:

```
Scheduler → Read TASK.md
  → Read knowledge/ (accumulated state)
  → Read DELIVERABLE.md + steering.md + feedback.md (ADR-149)
  → Step 1: UPDATE KNOWLEDGE
      Agent reads existing knowledge files as primary context
      Agent researches new signals (web, platforms)
      Agent WRITES BACK to knowledge/ (update entity files, add signals)
      Agent outputs a changelog of what changed
  → Step 2: DERIVE OUTPUT
      Agent reads ALL knowledge files (now updated)
      Agent produces the deliverable per DELIVERABLE.md
      Emphasis on diffs/changes — what's new since last cycle
  → Render assets → Compose HTML → Save to outputs/ → Deliver
```

**Critical new capability**: The agent writes to `knowledge/` during execution. This is the accumulation mechanism. The pipeline's headless tool set already includes `WriteWorkspace` — this extends it to task workspace via `TaskWorkspace.write()`.

For single-step tasks (no multi-step process), the knowledge update and output derivation happen in a single generation pass. The prompt instructs: "First update your knowledge files, then produce the output."

The write-back is scoped: agents can write to `/tasks/{slug}/knowledge/` during execution, but not to `/tasks/{slug}/TASK.md` or other task files. This preserves the charter/memory separation (ADR-149).

### Knowledge Scaffold at Task Creation

When a task is created from a type with `knowledge_schema`:
1. Create `knowledge/` folder
2. Write synthesis file from template (e.g., `landscape.md`)
3. If task `focus` specifies known entities, create initial entity files from template
4. Otherwise, the first run discovers and creates entity files

```python
# In handle_create_task():
if knowledge_schema:
    synthesis_file = knowledge_schema.get("synthesis_file")
    if synthesis_file:
        synthesis_template = knowledge_schema.get("synthesis_template", "")
        await tw.write(f"knowledge/{synthesis_file}", synthesis_template)
    
    # If focus names specific entities, create starter files
    if focus:
        entities = _parse_entities_from_focus(focus)  # "Acme Corp, Globex" → ["acme-corp", "globex"]
        for entity_slug, entity_name in entities:
            entity_content = knowledge_schema["entity_template"].format(entity_name=entity_name)
            folder = knowledge_schema.get("entity_folder", "")
            path = f"knowledge/{folder}/{entity_slug}.md" if folder else f"knowledge/{entity_slug}.md"
            await tw.write(path, entity_content)
```

### Diff-Aware Output Prompting

The output derivation step needs diff-awareness. The DELIVERABLE.md spec gains a section:

```markdown
## Derivative Mode
- This output is a VIEW over accumulated knowledge, not a standalone report
- Emphasize what CHANGED since last cycle
- Assume the reader has seen prior reports — don't repeat stable context
- Structure: What's New → What Changed → What It Means → Standing Context (brief)
```

For recurring tasks, each cycle's output should read like a briefing from an analyst who's been tracking continuously — not a researcher who just discovered the space. This is the "job replacement" framing: a human analyst doesn't write a fresh report from scratch every week.

### Mode Interaction with Knowledge

**Recurring**: Knowledge grows indefinitely. Each cycle adds signals, updates assessments, discovers new entities. The knowledge/ folder is the enduring value. Outputs are disposable digests.

**Goal**: Knowledge fills in toward completion. Each cycle deepens understanding of specific areas (financial analysis, risk assessment, etc.). When all knowledge areas meet a threshold, the goal is met. The final output synthesizes all accumulated knowledge.

**Reactive**: Knowledge persists across triggers. A meeting prep task accumulates relationship knowledge across all meetings with a contact. Each trigger adds a pre-meeting research snapshot and benefits from all prior accumulated context.

### Relationship to Agent Knowledge

Task knowledge (`/tasks/{slug}/knowledge/`) is separate from agent knowledge (`/agents/{slug}/`):

- **Agent knowledge** = methodology, preferences, reflections — HOW the agent works. Persists across all tasks the agent handles.
- **Task knowledge** = domain intelligence — WHAT the agent knows about this specific domain. Persists across all runs of this task.

An agent might handle multiple competitive intel tasks (different industries). Each task has its own knowledge substrate. The agent's methodology for competitive research is shared across all, but the per-competitor intelligence is task-scoped.

## Phases

### Phase 1: Knowledge Schema + Scaffold
- Add `knowledge_schema` to task type registry entries that benefit from accumulation
- Scaffold `knowledge/` folder + synthesis file + entity files at task creation
- Update `build_task_md_from_type()` to reflect knowledge-aware process steps
- Update workspace-conventions.md with knowledge/ folder conventions

### Phase 2: Pipeline Reads Knowledge
- `gather_task_context()` reads `knowledge/` files as primary context (before platform/web search)
- Knowledge files injected into agent prompt as "## Your Accumulated Knowledge"
- Existing runs still work (tasks without knowledge_schema skip this)

### Phase 3: Agent Write-Back to Knowledge
- Extend headless tool set: agent can write to `/tasks/{slug}/knowledge/` during execution
- Scope: only `knowledge/` subfolder, not task charter or memory
- "update-knowledge" step type: agent reads existing knowledge → researches → writes updates
- Agent outputs a changelog for the next step

### Phase 4: Diff-Aware Output Derivation
- Update process step instructions for "derive-output" steps
- DELIVERABLE.md gains "## Derivative Mode" section
- Output prompting emphasizes diffs: what changed, what's new, what it means
- Changelog from knowledge-update step injected as context for derivation step

### Phase 5: Knowledge-Aware TP Evaluation (Evolves ADR-149 Phase 3)
- ManageTask `evaluate` reads knowledge/ health: entity coverage, signal recency, depth
- TP can steer knowledge gaps: "next cycle, focus on Acme Corp — stale since March 15"
- Knowledge health signals in TP working memory for heartbeat evaluation

## Consequences

### Positive
- Tasks compound in value — the canonical product promise, now architecturally real
- The knowledge substrate IS the moat — accumulated intelligence that can't be replicated by starting over
- Output quality improves with tenure without any user intervention
- Agents operate at higher information hierarchy levels (L2→L3) as knowledge accumulates
- Task knowledge survives agent reassignment — the intelligence belongs to the task, not the worker
- Every mode benefits: recurring (continuous deepening), goal (structured coverage), reactive (cross-trigger memory)

### Negative
- More storage per task (knowledge files accumulate)
- Pipeline complexity: agent write-back during execution is a new capability
- Knowledge file quality depends on agent judgment — garbage in, garbage out
- Stale knowledge: entity files that stop being updated need cleanup

### Risks
- Knowledge files could grow unbounded. Mitigation: per-entity file size caps, signal log rotation (keep last N entries).
- Agent writes could corrupt knowledge files. Mitigation: write-back scoped to knowledge/ only, not charter/memory. Filesystem versioning (ADR-119) provides rollback.
- Knowledge structure too rigid per type. Mitigation: schemas are templates, not constraints — agents can create files outside the schema.
