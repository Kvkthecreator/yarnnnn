# ADR-185: Distribution Derivatives — Rich Post-Pipeline Repackaging

> **Status**: Proposed
> **Date**: 2026-04-15
> **Related**: ADR-148 (Output Artifact Architecture — RepurposeOutput primitive), ADR-170 (Compose Substrate — section kinds, surface types), ADR-177 (Section Kind Rendering — compose pipeline), ADR-174 (CONVENTIONS.md — workspace-wide rules), ADR-149 (Task Lifecycle — DELIVERABLE.md quality contract), ADR-181 (Feedback Layer — source-agnostic actuation), ADR-169 (MCP Context Hub — cross-LLM surface), ADR-182 (Pre-Gather — pipeline cost model)

## Context

YARNNN's compose substrate (ADR-170/177) produces rich multi-section HTML documents with structured data renderers (metric-cards, trend-charts, entity-grids, comparison-tables), embedded assets (matplotlib PNGs, mermaid diagrams, fetched images), and surface-type-aware layouts. This is where YARNNN out-competes incumbents.

But a 12-section dashboard with embedded charts isn't what gets shared on LinkedIn. The user needs their output repackaged for distribution — LinkedIn, Twitter/X, Medium, executive summary, slide deck — in forms that perform on those channels. A LinkedIn post with a hero chart gets 2-3x engagement over text-only.

ADR-148 Phase 4 implemented `RepurposeOutput` as a chat-mode primitive with two paths: mechanical (PDF/XLSX/DOCX via render service) and editorial (LinkedIn/Twitter/Medium via Haiku text adaptation). The backend works. But:

1. **Frontend is completely absent** — no UI to trigger, no display of results, `SHOW_REPURPOSE` ui_action unrecognized
2. **TP doesn't know about it** — `tools.py` never mentions RepurposeOutput
3. **Text-only editorial path is insufficient** — LinkedIn/Twitter perform better with visual assets, and the first-order output already produced those assets
4. **Manual invocation per-target per-run** — no automatic production, no workspace-level defaults
5. **MCP doesn't expose derivatives** — cross-LLM distribution (the strongest positioning) is blocked

The agent-native product thesis (doc 09) identifies distribution as the user's primary bottleneck. The architecture should produce distribution-ready derivatives automatically, rich with assets, ready to copy or pull via MCP.

## Decision

### Core model: derivatives are a post-pipeline step that repackages first-order output

Distribution derivatives are **not** separate tasks, not child tasks, not a new primitive. They are a pipeline post-step that runs after `_compose_and_persist()` on every `produces_deliverable` task that declares distribution targets.

```
Research context ──→ LLM generation ──→ output.md (first order)
                                            │
                                    ┌───────┴───────┐
                                    │               │
                              compose pipeline    distribution step
                                    │               │
                              output.html      distribution/{target}.html
                           (rich document)     (rich channel-optimized)
```

Both the compose pipeline and the distribution step consume `output.md` as their primary input. But the distribution step also reads the **asset inventory** from the compose step's `sys_manifest.json` to reuse generated assets.

### Two artifact classes, one production run

| | Primary deliverable | Distribution derivative |
|---|---|---|
| **Source** | `output.md` + accumulated context | `output.md` + asset inventory from `sys_manifest.json` |
| **Compose** | Full `page_structure`, multi-section, section-kind dispatch | Channel template, 2-4 slots (hero asset + text + CTA) |
| **Assets** | Generated by RuntimeDispatch during first-order step | Reused from first-order. Generated only when evaluation justifies. |
| **Output** | Rich HTML document (`output.html`) | Rich HTML card or text (`distribution/{target}.html` or `.md`) |
| **Surface** | iframe in DeliverableMiddle, delivered via email | Collapsible section below output, copy-first UX |
| **Lifecycle** | Versioned, evaluated, delivered | Ephemeral per-run, not separately evaluated |

### Graduated cost evaluation per target

The distribution step does **not** hardcode "lightweight." It evaluates per-target what's needed and routes accordingly:

**Tier 1 — Reuse** (compose only, ~$0.003):
Asset exists in manifest, fits channel requirements. Haiku writes hook/CTA text. Compose via channel template with embedded asset URL. Most common case.

**Tier 2 — Transform** (render service, ~$0.01):
Asset exists but needs mechanical transformation — wrong aspect ratio, wrong format (HTML table → PNG for social), needs cropping. Render service performs transformation. Then compose.

**Tier 3 — Generate** (RuntimeDispatch, ~$0.02-0.05):
No relevant asset exists. Two sub-paths:
- **Automatic**: When the channel target is declared in DELIVERABLE.md with explicit asset guidance (e.g., "LinkedIn: use hero stat card"), the pipeline generates via RuntimeDispatch. This is a conscious cost the user opted into by declaring the target.
- **TP-directed**: When TP or user explicitly requests "make me an Instagram card with the key stat," TP directs RuntimeDispatch. Full judgment, full cost.
- **Fallback**: When no asset exists and no generation is justified (undeclared target, no user intent), produce text-only derivative + write diagnostic feedback to `feedback.md`.

**Tier 0 — Text-only** (~$0.002):
Targets that are inherently text-only (executive summary, email teaser). No asset evaluation needed. Haiku adaptation of `output.md` directly.

The graduated model means: the system is *capable* of the full path when the situation demands it. The common case is cheap. The expensive case is opt-in.

### Asset gap diagnosis feeds back to task evolution

When the distribution step encounters a gap (no suitable asset for a declared target), it writes a feedback entry:

```markdown
### [2026-04-15] Distribution gap: linkedin
**Source:** system-distribution
No social-ready visual asset found for LinkedIn target. Available assets: [trend-chart.png (landscape, 1200x400)]. LinkedIn performs best with square or portrait hero images.
**Action:** Consider adding square stat-card asset to DELIVERABLE.md expected assets.
```

This follows the ADR-181 source-agnostic feedback model. The inference pipeline (`infer_task_deliverable_preferences()`) picks this up and evolves DELIVERABLE.md. Next cycle, the first-order output produces the right asset. The derivative layer gets richer over time without explicit user configuration.

### Declaration: workspace defaults + per-task override

**Workspace level** — `CONVENTIONS.md` gains a `### Distribution Defaults` section:

```markdown
### Distribution Defaults
<!-- Applied to all produces_deliverable tasks unless overridden. -->
- linkedin
- twitter
- summary
```

**Task level** — `DELIVERABLE.md` gains an optional `## Distribution` section:

```markdown
## Distribution
<!-- Overrides workspace defaults. Omit section to inherit. "none" to suppress. -->
- linkedin: Include pricing comparison chart as hero
- medium
```

**Resolution order**: Task `## Distribution` section → Workspace `### Distribution Defaults` → empty list (no derivatives).

Per-target guidance (e.g., "Include pricing comparison chart as hero") informs the asset evaluation at Tier 1-3. Without guidance, the step uses Haiku to select the most impactful asset from the manifest.

### Compose model: channel templates via existing render service

Distribution derivatives use the same `/compose` endpoint as the primary deliverable, but with **channel-specific surface types**:

| Channel | Surface type | Template shape |
|---------|-------------|----------------|
| `linkedin` | `social-card` | Hero asset (optional) + hook paragraph + body (150-250 words) + CTA |
| `twitter` | `thread-preview` | Per-tweet cards with optional inline asset on tweet 1 |
| `medium` | `article` | Standard article layout, same as `report` but with public framing |
| `summary` | `text-only` | No compose — raw markdown output, 150-250 words |
| `slides` | `deck` | Same compose as primary deliverable but restructured |

New surface types (`social-card`, `thread-preview`) are lightweight additions to the render service's compose vocabulary. They don't require new section-kind renderers — they compose from existing kinds (narrative, image-embed) in channel-shaped templates.

### Frontend: collapsible section below output, copy-first

Distribution derivatives surface as a collapsible section below the primary output in `DeliverableMiddle`, not as a separate tab. Tabs imply equivalence; derivatives are secondary.

```
[Output | History]

┌─ Latest output ──────────────────────────────────┐
│  [full iframe/markdown output as today]           │
└───────────────────────────────────────────────────┘

▸ Distribution (3 ready)                      [Copy all]
  ┌─ LinkedIn ──────────── 230 chars ──── [Copy] ──┐
  │  [rich preview: hook text + hero chart image]   │
  └─────────────────────────────────────────────────┘
  ┌─ Twitter ───────────── 5 tweets ───── [Copy] ──┐
  │  [thread preview with numbered tweets]          │
  └─────────────────────────────────────────────────┘
  ┌─ Summary ──────────── 180 words ───── [Copy] ──┐
  │  [three-paragraph text]                         │
  └─────────────────────────────────────────────────┘

[FeedbackStrip]
```

Each card shows a rich preview (including assets where available). Copy button copies the appropriate format: plain text for Twitter/summary, text + image URL for LinkedIn. Collapsed by default, badge shows count.

### MCP: derivatives are naturally addressable

`pull_context("linkedin post for competitive brief")` resolves to `/tasks/competitive-brief/outputs/{latest}/distribution/linkedin.md` via existing `QueryKnowledge` routing. No new MCP tools needed.

The `GET /api/tasks/{slug}/outputs/latest` endpoint gains an optional `include_distribution=true` parameter that returns derivative content alongside the primary output.

Cross-LLM workflow: user in ChatGPT says "post my competitive brief to LinkedIn" → ChatGPT calls `pull_context` via MCP → gets the rich derivative content → posts it using ChatGPT's own browser/API capability. **YARNNN produces, any LLM distributes.**

### RepurposeOutput primitive: scoped to export-only

The editorial targets (linkedin, twitter, medium, summary, slides) move out of `RepurposeOutput` into the pipeline distribution step. `RepurposeOutput` retains only mechanical export targets: **pdf, xlsx, docx, markdown**. These are format conversions of the composed HTML output — a different concern (export) from distribution (channel adaptation).

Consider renaming `RepurposeOutput` → `ExportOutput` for clarity. The primitive shrinks, the pipeline grows. No dual path.

## Consequences

### Positive
- Distribution derivatives produced automatically every run — no manual invocation
- Rich with assets (charts, images) reused from first-order output — not text-only
- Graduated cost evaluation prevents both wasteful generation and artificial ceilings
- Asset gap diagnosis drives task evolution — derivatives get richer over time
- Workspace-level defaults reduce per-task configuration burden
- MCP exposure enables cross-LLM distribution — strongest positioning
- Copy-first frontend UX matches editorial director workflow

### Negative
- Pipeline execution time increases (~2-5 seconds per target for Haiku + compose)
- Pipeline cost increases (~$0.003-0.01 per target per run, negligible vs. first-order ~$0.05-0.15)
- New surface types in render service require template implementation
- DELIVERABLE.md and CONVENTIONS.md gain new sections — TP must understand distribution declaration

### Neutral
- `RepurposeOutput` primitive shrinks to export-only (mechanical targets preserved)
- `sys_manifest.json` becomes load-bearing for asset inventory (already tracked, but now consumed by distribution step)
- Channel-specific editorial instructions move from hardcoded `repurpose.py` to DELIVERABLE.md / CONVENTIONS.md (user-editable, TP-steerable, inference-evolvable)

## Implementation Phases

### Phase 1: Pipeline integration + asset evaluation
- Add `_generate_distribution_derivatives()` post-step in `task_pipeline.py`
- Read distribution targets from DELIVERABLE.md → CONVENTIONS.md → empty
- Read asset inventory from `sys_manifest.json`
- Per-target evaluation: Tier 0 (text-only) and Tier 1 (reuse) paths
- Write derivatives to `/tasks/{slug}/outputs/{date}/distribution/{target}.md`
- Include distribution metadata in `sys_manifest.json`

### Phase 2: Channel templates in render service
- Add `social-card` and `thread-preview` surface types to compose vocabulary
- Lightweight channel templates that place existing assets + text
- Produce `.html` alongside `.md` for rich preview

### Phase 3: Frontend collapsible section
- Extend `GET /api/tasks/{slug}/outputs/latest` with distribution data
- Add collapsible Distribution section in `DeliverableMiddle.tsx`
- Copy button per target (text or text + asset URL)
- Badge count on collapsed state

### Phase 4: Tier 2-3 paths + feedback loop
- Render service transforms (crop, rasterize, resize) for asset format mismatch
- RuntimeDispatch path for missing assets when justified by declaration
- Text-only fallback + diagnostic feedback to `feedback.md` (ADR-181)
- Inference picks up distribution gap feedback → evolves DELIVERABLE.md

### Phase 5: MCP + workspace defaults
- Distribution-aware routing in `pull_context` (MCP composition layer)
- `### Distribution Defaults` in CONVENTIONS.md scaffold
- TP prompt guidance for distribution declaration

### Phase 6: RepurposeOutput scope reduction
- Remove editorial targets from `RepurposeOutput` primitive
- Retain mechanical targets (pdf, xlsx, docx, markdown) as export-only
- Consider rename to `ExportOutput`
- Update TP tool documentation
- Delete frontend `SHOW_REPURPOSE` dead code

## Supersedes

- ADR-148 Phase 4 editorial repurpose path (linkedin, twitter, medium, summary, slides targets move to pipeline)
- `EDITORIAL_TARGETS` and `EDITORIAL_INSTRUCTIONS` in `repurpose.py` (instructions move to DELIVERABLE.md / CONVENTIONS.md)

## Preserves

- ADR-148 Phase 4 mechanical repurpose path (pdf, xlsx, docx — export, not distribution)
- ADR-170/177 compose substrate (distribution uses same render service, different templates)
- ADR-181 feedback model (distribution gap diagnosis uses same source-agnostic layer)
- ADR-169 MCP three-tool surface (distribution files addressable via existing `pull_context`)
